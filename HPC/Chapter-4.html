Preliminary Concepts (Read Once)

If you write a parallel program with 4 CPUs, will it become 4× faster than the serial version? Usually NO — because of overheads (communication time, idle time, synchronization, extra work). This chapter teaches you the formulas used to measure and predict parallel program performance.

Key symbols used throughout this chapter

SymbolMeaning
pNumber of processors
nSize of the input (e.g. n numbers to sum, n×n matrix)
T1 or TserialTime taken by the best serial algorithm on 1 processor
Tp or TparallelTime taken by the parallel algorithm using p processors
SSpeedup (how many times faster)
EEfficiency (fraction of processor capacity used usefully)
ToTotal overhead time
fSerial fraction of the program (cannot be parallelized)

Q1 — Performance Metrics for Parallel Systems

Asked 4 times(May'23, May'24, May'25, Nov'25 — usually 6–7 marks)
How this question appeared in past papers:

(May'23, 6 marks) Explain different performance Metrics for Parallel Systems.

(May'24, 7 marks) Explain different performance Metrics for Parallel Systems.

(May'25, 6 marks) Explain performance matrices of parallel system.

(Nov'25, 4 marks) Explain performance matrices of parallel systems.

Answer
Suppose 1 worker can paint a wall in 8 hours. We hire 4 workers, expecting it to finish in 2 hours (= 8 ÷ 4). But because workers have to talk, share paint, and sometimes wait for each other, it actually takes 3 hours. So real speedup = 8/3 ≈ 2.67 (not the ideal 4). The "lost" 1.33× is coordination overhead. Performance metrics give us a precise way to measure this loss.

1. Speedup (S) — the most important metric

Speedup tells us how many times faster the parallel version is, compared to the best serial version.
S = Tserial / Tparallel = T1 / Tp
  • Ideal (linear) speedup: S = p. That means p processors make the program exactly p times faster.
  • Sub-linear speedup: S < p. Usual case in practice — overheads prevent us from reaching ideal.
  • Super-linear speedup: S > p. Rare. Caused mostly by cache effects — smaller chunks per processor fit better in cache.
  • Example: Tserial = 100 sec, Tparallel on 10 processors = 20 sec → Speedup = 5 (not the ideal 10).

2. Efficiency (E)

Efficiency tells us what fraction of a processor's capacity is being used for useful work (the rest is wasted on waiting, communication, etc.).
E = S / p = Tserial / (p · Tparallel)
  • Range: 0 < E ≤ 1 (usually expressed as a percentage).
  • E = 1 (100%) means every processor is fully utilized — ideal.
  • E = 0.6 (60%) means 40% of total processor time is wasted.
  • Example: speedup = 5 with 10 processors → E = 5/10 = 0.5 = 50%.
  • Generally, as p increases, E DECREASES (because overheads grow).

3. Cost (also called Work)

Cost is the total processor-time used by the parallel system, summed across all processors.
Cost = p × Tparallel
  • Think of cost as: "how much CPU-time did we BUY to solve this problem?"
  • Cost-Optimal algorithm: A parallel algorithm is called cost-optimal if its cost matches the best serial time: p · Tp = Θ(Tserial).
  • Cost-optimal means no wastage of total processor-seconds compared to serial.

4. Total Overhead (To)

Total overhead = the extra processor-time used by the parallel version that the serial version did not need.
To = p · Tparallel − Tserial
  • Overhead comes from communication, idling, synchronization, redundant work.
  • Smaller overhead → higher speedup and efficiency.

5. Redundancy

Redundancy = how much extra computation the parallel version does compared to the serial version.
Redundancy = (Total ops in parallel run) / (Total ops in serial run)
  • Redundancy ≥ 1 always.
  • Redundancy = 1 means parallel version does exactly the same work as serial.
  • Redundancy > 1 means duplicate or extra computation.

6. Utilization

Fraction of total available processor capacity used to do real (non-overhead) work.
Utilization = (Useful operations) / (Total processor-time available)

7. Execution Rate (Throughput)

  • Number of operations completed per unit time.
  • Often measured in FLOPS (floating-point operations per second).
  • For a parallel system: Execution rate = (Total operations) / (Wall-clock time).

8. Summary table of all metrics

MetricFormulaWhat it measures
Speedup (S)T1 / TpHow many times faster
Efficiency (E)S / pUseful work per processor
Costp × TpTotal processor-time used
Overhead (To)p · Tp − T1Wasted processor-time
RedundancyParallel ops / Serial opsExtra computation done
ThroughputTotal ops / TimeSpeed of useful computation

Key points to remember

  • Speedup S = T1 / Tp. Ideal = p.
  • Efficiency E = S / p. Ideal = 1 (100%).
  • Cost = p × Tp. Cost-optimal if = Tserial.
  • Overhead To = p · Tp − Tserial.
  • As p increases, E usually decreases due to overheads.
  • Super-linear speedup is rare and caused by cache effects.

Q2 — Parallel Matrix-Matrix Multiplication

Asked 4–5 times(May'23, May'24, May'25 twice, Nov'25 — usually 6–7 marks)
How this question appeared in past papers:

(May'23, 7 marks) Explain parallel Matrix-Matrix multiplication algorithm with example.

(May'24, 6 marks) Explain parallel Matrix-Matrix multiplication algorithm with example.

(May'25, 6 marks) Explain Matrix-Matrix multiplication in details.

(Nov'25, 7 marks) Explain parallel Matrix-Matrix multiplication algorithm with example.

Answer
Multiplying two n × n matrices A and B serially takes about n³ multiplications. For n = 1000, that is a billion operations — slow on one CPU. If we split this work cleverly across p processors, each processor handles n³/p operations and finishes much faster. The trick is to do this WITHOUT processors sending huge amounts of data to each other. Cannon's algorithm is one such clever way.

1. Definition

Matrix-Matrix Multiplication computes C = A × B, where A is n × n, B is n × n, and C is the resulting n × n matrix. Each element C[i][j] = Σ A[i][k] × B[k][j] for k = 0 to n − 1.

For each cell of C, we take one row from A and one column from B, multiply them element-wise, and add up the products. We do this for all n² cells of C.

2. Why parallelize?

  • Serial complexity = O(n³). For n = 1000, that is 10⁹ operations.
  • Each output cell C[i][j] can be computed independently of the others → great for parallelization.
  • Real-world use: deep learning (huge matrix multiplications), scientific simulations, graphics rendering.

3. Simple parallel approach — Row-wise 1D Block Partition

  • Partition matrix C (and A) row-wise among p processors. Each processor gets n/p rows.
  • Every processor needs the entire matrix B → so we broadcast B at the start.
  • Each processor independently computes its n/p rows of C.

Algorithm

1. Broadcast matrix B to all processors.
2. Scatter matrix A row-wise: processor i gets rows i·(n/p) to (i+1)·(n/p) - 1.
3. Each processor computes its assigned rows of C:
       for i in my_rows:
           for j = 0 to n - 1:
               C[i][j] = 0
               for k = 0 to n - 1:
                   C[i][j] += A[i][k] * B[k][j]
4. Gather the result rows of C to the root.

Complexity

  • Computation per processor = O(n³ / p).
  • Communication = O(n²) to broadcast B.
  • Speedup ≈ p when n is large compared to p.

4. Cannon's Algorithm (the famous 2D block algorithm)

Cannon's algorithm uses a 2D grid of processors and circular shifts to avoid the need to broadcast all of B. This makes it very memory-efficient.

Setup

  • Use a √p × √p grid of processors.
  • Partition A into √p × √p blocks, each of size n/√p × n/√p.
  • Same for B and C. Block (i, j) of each matrix is stored on processor (i, j).

Initial Alignment (one-time setup)

  • Shift each row i of matrix A LEFT by i positions (cyclic / circular shift).
  • Shift each column j of matrix B UP by j positions (cyclic shift).
  • After alignment, processor (i, j) holds A[i][(j+i) mod √p] and B[(i+j) mod √p][j], which can be multiplied together.

Main iteration (repeat √p times)

for step = 1 to √p:
    1. Each processor multiplies its current A-block by its current B-block,
       and adds the result to its local C-block.
    2. Shift A-blocks one position to the LEFT (cyclic, along rows).
    3. Shift B-blocks one position UP (cyclic, along columns).
end

Complexity of Cannon

  • Computation per processor per step = O((n/√p)³).
  • √p steps total → total computation per processor = O(n³ / p).
  • Communication per step = O(n² / p) for two shifts.
  • Total communication = O(n² / √p).
  • Memory per processor = O(n² / p) — much less than the simple 1D approach.

5. Fox's Algorithm (alternative to Cannon)

  • Same setup as Cannon (2D processor grid, block partition).
  • Uses broadcasts along rows instead of shifts for matrix A.
  • B-blocks still shift up cyclically.
  • Slightly higher memory because each processor temporarily holds the broadcast row.

6. DNS Algorithm (3D, for systems with many processors)

  • Uses p = n³ processors arranged in a 3D mesh.
  • Each processor computes exactly ONE multiplication.
  • Final addition done via a reduction along the third dimension.
  • Very fast: O(log n) time. But requires huge number of processors.

7. Comparison of algorithms

AlgorithmProcessorsTimeMemory/procCommunication
Simple 1DpO(n³/p) + O(n²)O(n²)Broadcast B
Cannon (2D)pO(n³/p) + O(n²/√p)O(n²/p)Shifts
Fox (2D)pO(n³/p) + O(n²/√p)O(n²/√p)Broadcast rows
DNS (3D)n³O(log n)O(1)Reduction along dim

8. Applications

  • Deep learning — fully-connected layers, attention mechanisms.
  • Scientific computing — finite element analysis, computational fluid dynamics.
  • Image processing — affine transformations, convolutions as matrix multiplication.
  • Graph algorithms — all-pairs shortest paths via matrix multiplication.
  • Computer graphics — 3D transformations, ray tracing.

Key points to remember

  • Serial complexity = O(n³); parallel = O(n³/p).
  • Simple 1D: broadcast B, scatter rows of A.
  • Cannon: 2D grid + circular shifts; memory-efficient.
  • Cannon alignment: shift A rows LEFT by row index, B columns UP by column index.
  • Cannon iteration: multiply, shift A left, shift B up, repeat √p times.
  • Fox uses broadcasts; DNS uses 3D with n³ processors for O(log n) time.

Q3 — Granularity and its Effects on Performance

Asked 3 times(May'23, May'24, Nov'25 — usually 6–7 marks)
How this question appeared in past papers:

(May'23 & Nov'25, 7 marks) What is granularity? What are effects of granularity on performance of parallel systems?

(May'24, 6 marks) Show effect of granularity on performance with addition of n numbers on p processing elements.

Answer
Imagine 100 cooks in a kitchen and 1000 potatoes to peel. Option A: give each cook 10 potatoes — fine. Option B: give 1 potato each to 1000 cooks (we only have 100, so 900 stand idle) — silly. Option C: give 1 potato to each of 100 cooks at a time — finished super fast but massive coordination overhead. Granularity is the SIZE of the work-chunk given to each processor.

1. Definition

Granularity is the size of the computation task assigned to each processor between successive communication or synchronization points. It tells us how much work each processor does on its own before having to talk to another processor.

2. Two extremes

Fine-grained parallelism

  • Each task is very small (few operations).
  • Many tasks per processor; frequent communication.
  • High communication overhead.
  • Useful only when the network is very fast (shared-memory, GPUs).
  • Example: each thread does just 1 addition.

Coarse-grained parallelism

  • Each task is large (many operations).
  • Few tasks per processor; infrequent communication.
  • Low communication overhead, but possible load imbalance.
  • Better for distributed-memory systems with slow networks.
  • Example: each processor handles a 1000-element sub-array independently.

3. Effects of granularity on performance

  • Fine granularity: good parallelism but high overhead → low efficiency.
  • Coarse granularity: low overhead but may not use all processors → also low speedup.
  • There is an optimal granularity that balances both.
  • Increasing granularity (more work per processor, fewer processors):
    • Reduces communication overhead.
    • Reduces idling overhead.
    • Usually improves efficiency, even if wall-clock time doesn't drop.

4. Detailed example — Adding n numbers on p processing elements

Case 1: Fine-grained (p = n, one number per processor)

  • Each processor holds 1 number.
  • Add them in log p steps using a parallel reduction tree.
  • Computation per step: 1 addition. Communication per step: send one number.
  • Time: Tp ≈ log n × (tc + ts + tw).
  • Most of the time is wasted on communication — efficiency is poor.

Case 2: Coarse-grained (p < n, n/p numbers per processor)

  • Each processor first locally sums its n/p numbers — NO communication needed.
  • Then a log p-step reduction tree combines partial sums.
  • Time: Tp ≈ (n/p) + log p (in units of one addition).
  • Communication is now only log p steps instead of log n.
  • Speedup ≈ n / ((n/p) + log p) → close to p when n is large.
  • Much better efficiency than the fine-grained case.

5. Granularity vs efficiency graph (typical shape)

Efficiency
   1.0 |     ____________
       |    /            \____
   0.5 |   /                  \____
       |  /                        \___
   0.0 +--|----|----|----|----|----|---→  Granularity
        very     optimal           very
        fine                       coarse

6. Factors that determine optimal granularity

  • Network speed — faster networks tolerate finer granularity.
  • Problem size n — larger n allows coarser granularity.
  • Number of processors p — more processors means each gets less work.
  • Synchronization frequency — fewer sync points allow coarser granularity.
  • Load distribution — if work is uneven, fine granularity helps balance.

Key points to remember

  • Granularity = amount of work per processor between communications.
  • Fine-grained → high parallelism, high overhead → low efficiency.
  • Coarse-grained → low overhead, possible load imbalance.
  • Optimal granularity = balance between the two.
  • For adding n numbers on p processors: coarse-grained gives Tp ≈ n/p + log p.

Q4 — Sources of Overhead in Parallel Programs

Asked 3 times(May'23, May'24, Nov'25 — usually 6–7 marks)
How this question appeared in past papers:

(May'23, 6 marks) Explain various sources of overhead in parallel systems.

(May'24, 7 marks) Explain various sources of overhead in parallel systems.

(Nov'25, 6 marks) Explain sources of overhead in parallel program.

Answer
A serial program is like ONE chef cooking one meal — no waste of effort. A parallel program is like 10 chefs in one kitchen cooking the same meal together — some chefs wait for ingredients, some stand idle when their dish is done, some accidentally cook the same thing twice. All that lost time is OVERHEAD.

1. Definition

The overhead in a parallel program is the extra processor-time used by the parallel version that the serial version did not need.

Toverhead (To) = p · Tparallel − Tserial

This overhead is the reason real speedup is always less than ideal (p).

2. The 4 main sources of overhead

Source 1: Inter-Process Communication (usually the biggest)

  • Time spent sending and receiving messages between processors.
  • Includes startup time (ts) and per-word transmit time (tw) for each message.
  • Becomes worse on slow networks or with many small messages.
  • Grows with p (more processors talking to each other).

Source 2: Idling (Load Imbalance + Synchronization Waits)

  • A processor sits idle because:
    • Load imbalance: some processors finish their work; others still busy.
    • Synchronization: all must wait at a barrier until the slowest arrives.
    • Serial sections: some part of the code runs only on one processor; others wait.
  • Common in iterative solvers and algorithms with strict synchronization.

Source 3: Excess Computation

  • Parallel algorithm sometimes does MORE total work than the serial one because:
    • Best serial algorithm is hard to parallelize; we use a slightly worse but more parallel one.
    • Processors recompute values locally instead of communicating them.
    • Duplicate or wasted work due to speculation.

Source 4: Extra Memory / Storage

  • Each processor often keeps duplicate copies of some data (e.g., the full matrix B).
  • Total memory used across all processors > memory of serial version.
  • Can slow things down due to cache misses or paging.

3. Other minor sources

  • Contention for shared resources (shared memory bus, disk, printer).
  • Overhead of fork/join (creating and joining threads or processes).
  • Scheduling overhead (OS context switches).
  • Network congestion when many messages cross at the same time.

4. How overhead affects key metrics

  • Higher overhead → higher Tp → lower speedup.
  • Higher overhead → higher cost (p · Tp) → lower efficiency.
  • Goal: minimize To by balancing computation, communication, and synchronization.

5. Summary table

SourceWhat it isHow to reduce
CommunicationTime spent in send/receiveCombine messages, better topology, overlap with computation
Idling / Load imbalanceWaiting for work or syncBetter load balancing, dynamic scheduling
SynchronizationWaiting at barriersFewer barriers, asynchronous algorithms
Excess computationExtra work vs serialUse more efficient parallel algorithm
Extra memoryDuplicate copiesUse shared memory, better partition
ContentionFighting for resourcesReplicate resource, use locks carefully

Key points to remember

  • 4 main sources: communication, idling, excess computation, extra memory.
  • Communication is usually the biggest overhead.
  • Idling caused by load imbalance, synchronization, or serial sections.
  • To = p · Tp − Tserial.
  • Lower overhead → higher efficiency and speedup.

Q5 — Minimum Execution Time and Cost-Optimal Execution Time

Asked 2 times(May'23, May'25 — usually 4–5 marks)
How this question appeared in past papers:

(May'23, 4 marks) Explain Minimum Execution Time and Minimum Cost Optimal Execution Time.

(May'25, 5 marks) Write a note on minimum and cost optimal execution time.

Answer
Hiring 1000 workers might let you finish a project in 1 day, but you pay for 1000 worker-days. Hiring 50 workers might take 5 days but cost 250 worker-days — much cheaper. There is a sweet spot between "fastest" and "most efficient". This is exactly what minimum execution time vs cost-optimal execution time capture.

1. Minimum Execution Time (Tmin)

The shortest possible time to solve the problem on a parallel system, when we can use as many processors as we want.
  • To find it, take the parallel time formula Tp(n, p), treat p as a variable, and find the p that minimizes Tp.
  • Mathematically: solve d Tp / dp = 0.
  • Example: For adding n numbers, Tp ≈ n/p + log p.
    • Derivative: −n/p² + 1/(p ln 2) = 0 → p ≈ n / ln 2.
    • So Tmin = Θ(log n) using p ≈ n processors.
  • Reaching Tmin often requires many processors → poor efficiency.

2. Cost-Optimal Execution Time (Tpcost-opt)

The shortest time we can achieve while still keeping the algorithm cost-optimal (no wasted processor-time vs serial).
  • Cost-optimal condition: p · Tp = Θ(Tserial).
  • Means efficiency stays constant (does not vanish as p grows).
  • Usually Tpcost-opt > Tmin: accept slightly longer time for better efficiency.
  • Example: For adding n numbers, cost-optimal when p log p ≤ n.
    • Setting p = n / log n gives Tpcost-opt = Θ(log n).
    • Same asymptotic time as Tmin, but uses fewer processors.

3. Comparison

MetricGoalTrade-off
TminFinish as fast as possibleUses many processors, low efficiency
Tpcost-optKeep efficiency reasonableUses fewer processors, slightly slower

4. Numerical example (adding n = 1024 numbers)

pTp ≈ n/p + log pCost = p·TpEfficiency
110241024100%
3232 + 5 = 371184~86%
1288 + 7 = 151920~53%
10241 + 10 = 11 (Tmin)11264~9%

At p = 128 we get close to Tmin but with 6× better efficiency than at p = 1024. That is the cost-optimal sweet spot.

Key points to remember

  • Tmin = shortest possible parallel time, with many processors.
  • Found by minimizing Tp(n, p) over p.
  • Tpcost-opt = shortest time keeping cost optimal.
  • Cost-optimal uses fewer processors than Tmin.
  • Real systems prefer cost-optimal over absolute minimum.

Q6 — Amdahl's Law and Gustafson's Law

Asked 1 time(May'24 — 4 marks)
How this question appeared in past papers:

(May'24, 4 marks) Explain Amdahl's and Gustafson's law.

Answer
Suppose 80% of a project can be done in parallel, but 20% MUST be done by one person (e.g., final review by the manager). Even with 1000 helpers, that 20% takes the same time. Amdahl's law says this serial part puts a hard cap on speedup. Gustafson's law says: as we add more helpers, we usually scale up the project size too — parallel part grows, serial stays small, speedup keeps growing.

1. Amdahl's Law (for FIXED problem size)

Theoretical maximum speedup when only a fraction of the program is parallelizable.

Let f = fraction that is serial; (1 − f) = fraction that is parallel.

Speedupmax = 1 / [ f + (1 − f) / p ]
  • As p → ∞, the formula approaches 1/f. Upper bound on speedup.
  • If f = 0.10 (10% serial), max speedup = 10 — even with 1000 processors!
  • If f = 0.05, max speedup = 20.
  • Lesson: small serial portions severely limit speedup.

Example

  • f = 0.2 (20% serial), p = 10.
  • Speedup = 1 / [0.2 + 0.8/10] = 1 / 0.28 ≈ 3.57.
  • With p = ∞: Speedup = 1/0.2 = 5. So infinite processors give only 5× speedup.

2. Gustafson's Law (for SCALED problem size)

As we add more processors, the problem size also grows. The parallel portion grows linearly with p, serial portion stays the same.

Speedup = p − f · (p − 1)
  • Here f is the fraction of the SCALED-UP problem that is serial.
  • As p increases, speedup grows almost linearly with p.
  • Lesson: For big-data, parallelism continues to pay off.

Example

  • f = 0.05, p = 10.
  • Speedup = 10 − 0.05 × 9 = 9.55. Almost ideal!

3. Amdahl vs Gustafson — side-by-side

AspectAmdahl's LawGustafson's Law
Problem sizeFixedScales with p
ViewPessimisticOptimistic
Max speedupBounded by 1/fGrows linearly with p
Best forSmall or fixed problemsBig-data, scientific HPC
Year proposed19671988

Key points to remember

  • Amdahl: Speedupmax = 1 / [f + (1−f)/p], capped at 1/f.
  • Gustafson: Speedup = p − f·(p−1), grows almost linearly.
  • Amdahl is pessimistic (fixed problem), Gustafson is optimistic (scaled problem).
  • Even 10% serial code limits Amdahl speedup to 10×.

Q7 — Scalability of Parallel Systems

Asked 1 time(May'24 — 4 marks)
How this question appeared in past papers:

(May'24, 4 marks) Comment on "Scalability of Parallel Systems".

Answer

1. Definition

Scalability = the ability of a parallel system to maintain (or improve) performance, particularly efficiency, as both the number of processors p and the problem size n increase.

A scalable system keeps working efficiently even when you make the problem bigger AND add more processors.

2. Two types of scalability

Strong scaling

  • Keep n FIXED, increase p, see how Tp drops.
  • Related to Amdahl's law.
  • Hard to achieve; plateaus due to communication overhead.
  • Example: simulating a fixed-size weather grid with more CPUs.

Weak scaling

  • Increase p AND n proportionally; see if time stays roughly constant.
  • Related to Gustafson's law.
  • Easier to achieve; preferred in HPC.
  • Example: bigger weather grids on bigger machines.

3. Isoefficiency Function

Tells us how fast n must grow with p to KEEP efficiency constant.

  • Slow growth (linear in p) → highly scalable.
  • Fast growth (exponential) → poorly scalable.

4. Factors affecting scalability

  • Communication overhead — grows with p.
  • Memory per processor — limits per-processor work.
  • Network topology / bandwidth — hypercube scales better than ring.
  • Algorithm structure — logarithmic communication scales well.
  • Load balancing — uneven work distribution kills scalability.

Key points to remember

  • Scalability = maintain efficiency as p and n grow.
  • Strong scaling: fix n, increase p. Weak scaling: scale both.
  • Isoefficiency: how n must grow with p to keep E constant.
  • Communication is the main barrier to scalability.

Q8 — Scaling Down (Downsizing) a Parallel System

Asked 1 time(May'23 — 4 marks)
How this question appeared in past papers:

(May'23, 4 marks) Explain "Scaling Down (downsizing)" a parallel system with example.

Answer

1. Definition

Scaling Down (downsizing) = using FEWER processors than the maximum theoretically allowed, in order to achieve BETTER efficiency, even if wall-clock time goes up slightly.

Sometimes using fewer processors is better — each one stays busier, less time wasted on communication, total cost is lower.

2. Why scale down?

  • More processors → more communication and idling overhead.
  • Efficiency E drops as p increases.
  • For the same wall-clock time, fewer processors give better cost-per-result.
  • Using a million processors is often impractical.

3. Example — Adding 1024 numbers

pTp ≈ 1024/p + log pEfficiency E
1024 (max)1 + 10 = 119.1%
1288 + 7 = 1553%
3232 + 5 = 3786%

Going from 1024 down to 128 processors makes the run slightly slower (15 vs 11) but EFFICIENCY jumps from 9% to 53% — much better resource use.

4. When is scaling down a good idea?

  • When efficiency at max p is very low.
  • When the cost (p · Tp) is the main concern.
  • When the parallel infrastructure is shared with other jobs.
  • When the problem size n is small relative to p.

5. Trade-off

  • Scaling down INCREASES Tp slightly.
  • Scaling down DECREASES cost (p · Tp).
  • Scaling down INCREASES efficiency E.
  • Goal: find the sweet spot (cost-optimal point).

Key points to remember

  • Scaling down = using fewer processors than max for better efficiency.
  • More processors → more overhead → lower efficiency.
  • Wall-clock time goes up slightly; total cost goes down.
  • Sweet spot = cost-optimal number of processors.

Q9 — Matrix-Vector Multiplication (1D & 2D partitioning)

Asked 1 time(May'25 — 9 marks)
How this question appeared in past papers:

(May'25, 9 marks) Write a short note on Matrix vector multiplication using: i) Row-wise 1D partitioning ii) 2D partitioning iii) Comparison of 1D and 2D partitioning.

Answer
We want y = A · x where A is n × n and x is a vector of size n. Each element of y comes from multiplying a row of A by the vector x. We need to decide HOW to split this among p processors. Two main ways: cut A into rows (1D) or cut A into rectangular blocks (2D).

1. The Problem

Compute y[i] = Σ A[i][k] · x[k] for k = 0 to n − 1. Serial complexity: O(n²).

2. Row-wise 1D Partitioning

  • Each processor gets n/p consecutive rows of A.
  • Each processor needs the FULL vector x to compute its rows of y.
  • Vector x is replicated across all processors (broadcast at start).

Algorithm

1. Broadcast vector x to all p processors.
2. Scatter A row-wise: processor i gets rows i·(n/p) to (i+1)·(n/p) - 1.
3. Each processor computes its n/p elements of y:
       for i in my_rows:
           y[i] = 0
           for k = 0 to n - 1:
               y[i] += A[i][k] * x[k]
4. (Optional) gather y to the root.

Complexity

  • Computation per processor = O(n²/p).
  • Communication = O(n) for broadcasting x.
  • Maximum useful p ≈ n.

3. 2D Partitioning (block partitioning)

  • Arrange p processors as √p × √p grid.
  • Partition A into √p × √p blocks of size n/√p × n/√p.
  • x distributed in pieces — chunk j goes to all processors in column j.

Algorithm

1. Distribute vector x: split into √p chunks; chunk j to all processors in column j.
2. Each processor (i, j) holds an (n/√p)×(n/√p) block of A and chunk j of x.
3. Each processor computes partial: partial[i] = A_block × x_chunk
4. Within each ROW of the processor grid, do an all-to-one reduction:
       y_block[i] = Σ partial sums from all processors in row i
   Result on leftmost processor of each row.

Complexity

  • Computation per processor = O(n²/p).
  • Communication ≈ O(n/√p) per processor.
  • Memory per processor = O(n²/p + n/√p).
  • Maximum useful p ≈ n².

4. Comparison of 1D and 2D Partitioning

Feature1D Row Partition2D Partition
Communication per processorO(n)O(n/√p)
Memory per processorO(n²/p + n)O(n²/p + n/√p)
Max useful processorsnn²
ScalabilityLowerHigher
Complexity (programming)SimplerSlightly more complex
Best forSmall pLarge p, large matrices

Conclusion: 1D is simpler but limited (p ≤ n). 2D scales much better — preferred for large systems.

Key points to remember

  • 1D: scatter rows of A, broadcast x; simple but limited.
  • 2D: √p × √p grid, smaller per-processor communication.
  • 2D scales to more processors.
  • 1D communication = O(n); 2D = O(n/√p).
  • 2D needs row-wise reduction at end.

Q10 — Dense Matrix Algorithms

Asked 1 time(May'25 — 8 marks)
How this question appeared in past papers:

(May'25, 8 marks) Explain the Dense matrix algorithms: i) Matrix vector multiplication. ii) Matrix-Matrix multiplication.

Answer

1. What is a "dense" matrix?

A matrix where MOST elements are non-zero. (Opposite: "sparse" where most are zero.) Dense matrix algorithms touch every element, so time is O(n²) for matrix-vector and O(n³) for matrix-matrix.

2. Why we study them

  • Matrix operations are the building blocks of many scientific and engineering computations.
  • They have very regular structure → ideal for parallelism.
  • Used in linear systems (Ax = b), eigenvalue computation, FFT, deep learning.

3. Dense Matrix-Vector Multiplication (y = A · x)

See Q9 above for full algorithms. Two main schemes:

  • Row-wise 1D partition: each processor holds n/p rows of A and the full vector x.
  • 2D block partition: processors arranged in √p × √p grid; partial sums reduced row-wise.

Summary of costs

SchemeComputationCommunication
1D row partitionO(n²/p)O(n)
2D block partitionO(n²/p)O(n/√p)

4. Dense Matrix-Matrix Multiplication (C = A · B)

See Q2 above for full algorithms. Main approaches:

  • Simple 1D: broadcast B, scatter rows of A.
  • Cannon's algorithm: 2D grid + circular shifts; memory-efficient.
  • Fox's algorithm: 2D grid + broadcasts along rows.
  • DNS algorithm: 3D grid with n³ processors → O(log n) time.

Summary of costs

AlgorithmProcessorsComputationMemory/proc
Simple 1DpO(n³/p)O(n²)
Cannon 2DpO(n³/p)O(n²/p)
Fox 2DpO(n³/p)O(n²/√p)
DNS 3Dn³O(log n)O(1)

5. Common applications of dense matrix algorithms

  • Solving linear systems (Gaussian elimination, LU decomposition).
  • Eigenvalue and singular-value computations.
  • Deep learning — convolution layers, fully-connected layers.
  • Scientific simulations — finite element analysis, fluid dynamics.
  • Computer graphics — transformations.
  • Image processing — filters, PCA.

Key points to remember

  • Dense matrix = most elements non-zero.
  • Matrix-vector: O(n²) serial; parallelized via 1D row or 2D block.
  • Matrix-matrix: O(n³) serial; parallelized via 1D, Cannon, Fox, DNS.
  • Cannon is memory-efficient and widely used.
  • 2D and 3D scale to many more processors than 1D.

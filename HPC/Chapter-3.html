Preliminary Concepts (Must Know Before Anything)

Before we start the actual questions, let's understand 5 small ideas that will appear in EVERY answer. If you skip these, the formulas later will not make sense.

1. What is a "node" and what is a "topology"?

A "node" is just one computer (or processor) in our parallel system. A "topology" is the SHAPE in which we connect all these computers together using wires.

Think of "p" as the total number of nodes (= number of processors working together). The four shapes you must remember:

Topology (shape)How it looksEach node connects to"Diameter" (farthest distance)
Linear arrayComputers in a straight line1 or 2 neighboursp − 1
RingComputers in a circle (ends joined)2 neighboursp / 2
2-D MeshGrid (e.g. 3×3 square)2 to 4 neighbours2(√p − 1)
Hypercube3D cube (8 nodes = corners of cube)log p neighbourslog p

2. Hypercube — the most important topology in this chapter

  • A d-dimensional hypercube has p = 2d nodes. So d = log2 p.
  • Each node has a label which is a d-bit binary number (e.g. for 8 nodes, labels are 000, 001, 010, ..., 111).
  • Two nodes are directly connected if their labels differ in exactly 1 bit. For example, 010 and 011 differ in only the last bit, so they are connected.
  • Each node has exactly d = log p neighbours.
  • Why it is fast: because every node has many neighbours, data can spread very quickly (in only log p steps, not p steps).
8-node Hypercube (3-cube):

       110────────111
       ╱│         ╱│
      ╱ │        ╱ │
    010────────011 │
     │ 100──────┼─101
     │ ╱        │ ╱
     │╱         │╱
    000────────001

3. Two cost numbers — ts and tw

Every time one computer sends a message to another, the time taken has TWO parts: a fixed "setup time" plus an amount that depends on the size of the message.
  • ts (startup time) — fixed cost to prepare and start sending a message (handshake, header, etc.). Like dialling a phone number before you can start talking.
  • tw (per-word transfer time) — time taken to send 1 word of data on the wire. Like how fast you can talk once the call connects.
  • If a message has m words, the total time on one link is:
Cost of one message = ts + m · tw

4. What is "log p"? (a small math reminder)

"log p" means log base 2 of p. The trick is that log p is a very small number compared to p:

  • p = 2 → log p = 1
  • p = 8 → log p = 3
  • p = 64 → log p = 6
  • p = 1024 → log p = 10

So any algorithm whose time is "log p" is much faster than one whose time is "p". Hypercube algorithms are mostly log p — that is why we love hypercubes.

5. What is a "collective operation"?

A "collective operation" is a special function call that ALL processors must call at the same time. It performs a coordinated communication pattern (like broadcast, scatter, gather, reduce).

The library that gives us these functions is called MPI (Message Passing Interface). You will see MPI function names like MPI_Bcast, MPI_Scatter, MPI_Reduce throughout this chapter.

Q1 — One-to-All Broadcast and All-to-One Reduction

Asked 4 times(May'23, May'24, May'25, Nov'25 — usually 6–7 marks)
How this question appeared in past papers:

(May'23, 7 marks) Explain with diagram One-to-all broadcast on an eight-node ring with recursive doubling technique. Node 0 is the source of the broadcast. Also explain all-to-one reduction with node 0 as destination.

(May'24 & Nov'25, 7 marks) What is one-to-all broadcast? Explain it with the help of algorithm for one-to-all broadcast on hypercube. Comment on cost calculation.

(May'25, 6 marks) Explain Broadcast and Reduce operation with diagram. Explain All-to-One Broadcast and Reduction on a ring.

Answer
A school teacher (the source/root computer) has 1 printed notice. There are 30 students (the other computers) who all need a copy. Slow way: teacher hand-delivers to each student one by one (30 trips). Smart way: teacher gives copy to 1 student → now 2 of them give copies to 2 new students → now 4 give to 4 new → and so on. In just ⌈log2 30⌉ = 5 rounds, everyone has the notice. This "doubling every round" idea is called recursive doubling.

1. Definition

One-to-All Broadcast is a collective communication operation in which a single source computer (called the root) sends the same identical message to all other p − 1 computers in the parallel system.

ONE sender → ALL receivers, and everyone gets the SAME message.

2. Key characteristics

  • It is a one-to-many pattern.
  • The source computer is called the root.
  • All other p − 1 computers receive an identical copy of the same m-word message.
  • Naïve method (root sends to each non-root one by one) takes p − 1 steps — slow.
  • Smart method using recursive doubling takes only log p steps.
  • The reverse / dual operation is All-to-One Reduction.
  • In MPI, this is implemented by the function MPI_Bcast.
  • It is a fundamental building block used by many higher operations (all-to-all broadcast, all-reduce, prefix-sum).

3. The Recursive Doubling Idea (most important to write)

In every step, every computer that already has the message sends a copy to one computer that doesn't have it yet. So:

  • After step 1 — 2 computers have it
  • After step 2 — 4 computers have it
  • After step 3 — 8 computers have it
  • After step k — 2k computers have it

So for p nodes, total steps = ⌈log2 p⌉.

4. Algorithm — One-to-All Broadcast on a Hypercube

Algorithm ONE_TO_ALL_BCAST_HYPERCUBE (d, my_id, X):
   // d  = number of dimensions (= log p)
   // my_id = the label of this node (binary)
   // X = the message
   mask = 2^d − 1
   for i = d − 1 down to 0:
       mask = mask XOR 2^i        // clear bit i in mask
       if (my_id AND mask) == 0:  // node is active in this step
           if (my_id AND 2^i) == 0:
               partner = my_id XOR 2^i
               send X to partner
           else:
               partner = my_id XOR 2^i
               receive X from partner
   end for

0  1  2  3  4  5  6  7

●—○—○—○—●—○—○—○         Now {0, 4} have data

Step 2: Node 0 → Node 2, AND Node 4 → Node 6 (both in parallel)
●—○—●—○—●—○—●—○         Now {0, 2, 4, 6} have data

●—●—●—●—●—●—●—●         ALL 8 nodes have data — done in 3 steps ✓

6. Step-by-step on an 8-node Hypercube (3-cube, source = node 000)

  • Step 1 (dim 2): 000 → 100. Holders: {000, 100}.
  • Step 2 (dim 1): 000 → 010 and 100 → 110. Holders: {000, 010, 100, 110}.
  • Step 3 (dim 0): 000 → 001, 010 → 011, 100 → 101, 110 → 111. All 8 nodes have data.

7. Cost calculation (comment on it)

Each step transmits one m-word message across one link. So each step costs (ts + m · tw). Total steps = log p.

Tone-to-all = (ts + m · tw) · log p

Numerical idea:

  • For p = 8 → 3 steps
  • For p = 64 → 6 steps
  • For p = 1024 → 10 steps

Even for thousand-node systems, broadcast finishes in just 10 communication steps. This is why hypercube/recursive-doubling is so powerful.

8. Applications of One-to-All Broadcast (write 3–5)

  • Master computer broadcasting input data (matrix, vector) to all worker computers in parallel computations.
  • Broadcasting the pivot element in parallel quicksort.
  • Distributing common configuration / shared constants to all nodes in a cluster.
  • Broadcasting the search target in parallel search algorithms.
  • Sending model parameters to all GPU workers in distributed deep-learning training.

9. All-to-One Reduction (Part B of the question)

Reverse situation. Every student has written one number on paper. The teacher wants the TOTAL. Students pair up — each pair adds the two numbers and passes ONE paper forward. Pairs of pairs then add again. After ⌈log2 30⌉ = 5 rounds, the teacher receives one paper holding the final total.

Definition

Every computer holds a value (v0, v1, ..., vp−1). These values are combined using an associative operator (such as +, ×, max, min, AND, OR, XOR) into ONE single result at the destination computer. It is the EXACT REVERSE of one-to-all broadcast.

ALL senders → ONE receiver, with values combined (added / max'd / min'd) into one final value.

Key characteristics

  • It is a many-to-one communication pattern.
  • The operator must be associative so that grouping order does not matter (since pairs combine in parallel).
  • It runs in log p steps using recursive halving (the reverse of recursive doubling).
  • At every step the number of "active" nodes is halved.
  • In MPI it is implemented by MPI_Reduce.
  • Common MPI reduction operators: MPI_SUM, MPI_PROD, MPI_MAX, MPI_MIN, MPI_LAND, MPI_LOR, MPI_BAND, MPI_BOR.

Step-by-step on 8-node Ring (destination = Node 0)

Initial:    v0   v1   v2   v3   v4   v5   v6   v7
            P0   P1   P2   P3   P4   P5   P6   P7

Step 1: 1→0, 3→2, 5→4, 7→6   (receiver ADDS the value)
        [v0+v1]   [v2+v3]   [v4+v5]   [v6+v7]
           P0        P2        P4        P6

Step 2: 2→0, 6→4
        [v0+v1+v2+v3]         [v4+v5+v6+v7]
              P0                    P4

Step 3: 4→0
        [v0+v1+v2+v3+v4+v5+v6+v7]
                  P0     ✓ Destination has the final result

Cost of All-to-One Reduction

Treduction = (ts + m · tw) · log p

Applications of All-to-One Reduction

  • Computing the global sum / average / max / min from partial results of all workers.
  • Computing the dot product of two distributed vectors.
  • Convergence detection in iterative numerical solvers (sum of error from all processors).
  • Counting "hits" in Monte-Carlo simulations (e.g. Pi estimation).
  • Voting and consensus algorithms in distributed systems.

Key points to remember (write these for sure)

  • One-to-all broadcast = same message from one root to all others; uses recursive doubling.
  • All-to-one reduction = combine all values into one at destination; uses recursive halving.
  • Both take only log p steps on a hypercube/ring.
  • Cost formula for both: (ts + m · tw) · log p.
  • They are duals — same number of steps, just data flowing in opposite direction.
  • MPI calls: MPI_Bcast for broadcast and MPI_Reduce for reduction.
  • Reduction operator must be associative: +, ×, max, min, AND, OR, XOR.

Q2 — All-to-All Broadcast and Reduction

Asked 4 times(May'23, May'24, May'25, Nov'25 — usually 6–7 marks)
How this question appeared in past papers:

(May'23, 7 marks) What is all-to-all broadcast communication operation? Explain all-to-all broadcast on an eight-node ring with step-wise diagrams. (Show first two steps and last communication step.)

(May'24, 7 marks) Explain all-to-all broadcast and all-to-all reduction communication operation with example. Discuss cost analysis.

(May'25 & Nov'25) Explain with example and algorithm all-to-all broadcast on 3 × 3 mesh. Explain cost analysis of all-to-all broadcast operation.

Answer
Eight friends sit in a circle. Each one has a DIFFERENT juicy news to share. After the operation, every friend should know ALL 8 news items. Each person whispers their news to the friend on the right; in the next round, that friend forwards what was received; after 7 rounds, every friend has heard every piece of news.

1. Definition

All-to-All Broadcast (also called gossip or total exchange) is a collective communication operation in which every computer broadcasts its OWN message to every other computer, simultaneously. At the end, each computer has a copy of ALL p messages (one from every node, including itself).

EVERY sender → EVERY receiver, and each receiver collects everyone's different messages.

2. Key characteristics

  • It is a many-to-many pattern — every node acts as both sender and receiver.
  • Equivalent to running p simultaneous one-to-all broadcasts.
  • Total data received by each node = p · m words.
  • The number of steps depends on the topology:
    • Ring → p − 1 steps
    • 2-D Mesh (√p × √p) → 2(√p − 1) steps
    • Hypercube → log p steps
  • On a hypercube, the message size DOUBLES at every step (because at each step a node receives all the data its partner has).
  • The dual operation is All-to-All Reduction.
  • In MPI: MPI_Allgather for broadcast and MPI_Reduce_scatter for reduction.

3. Algorithm — All-to-All Broadcast on a Ring

Algorithm ALL_TO_ALL_BCAST_RING (my_id, my_msg, p):
   left  = (my_id - 1 + p) mod p
   right = (my_id + 1) mod p
   result[my_id] = my_msg
   msg = my_msg
   for i = 1 to p - 1:
       send msg to right
       receive msg from left
       result[(my_id - i + p) mod p] = msg
   end for

4. Step-by-step on an 8-node ring (each node forwards to the RIGHT)

Notation: Pi:{M0, M1} means processor i currently holds messages M0 and M1.

Initial:
P0:{M0}   P1:{M1}   P2:{M2}   P3:{M3}   P4:{M4}   P5:{M5}   P6:{M6}   P7:{M7}

Step 1:  Each processor sends its OWN message to right neighbour
P0:{M0,M7}  P1:{M1,M0}  P2:{M2,M1}  P3:{M3,M2}  ...  P7:{M7,M6}

Step 2:  Each forwards the most recently received message to its right neighbour
P0:{M0,M7,M6}  P1:{M1,M0,M7}  P2:{M2,M1,M0}  ...  P7:{M7,M6,M5}

(steps 3, 4, 5, 6 continue similarly ...)

Step 7 (LAST step): every node now holds all 8 messages.
P0 through P7 all hold {M0, M1, M2, M3, M4, M5, M6, M7}    ✓

5. All-to-All Broadcast on a 3 × 3 Mesh (two-phase algorithm)

The mesh algorithm has 2 phases — first along the rows, then along the columns.

Initial layout (Pij holds its own message Mij):
P00:{M00}  P01:{M01}  P02:{M02}
P10:{M10}  P11:{M11}  P12:{M12}
P20:{M20}  P21:{M21}  P22:{M22}

Phase 1 — Row-wise all-to-all broadcast (within each row of 3 nodes):
After this phase, every node holds all 3 messages of its row.
   Row 0 nodes now each hold {M00, M01, M02}
   Row 1 nodes now each hold {M10, M11, M12}
   Row 2 nodes now each hold {M20, M21, M22}

Phase 2 — Column-wise all-to-all broadcast (within each column):
Each node now sends a combined "row message" of size 3m.
After this phase, every node holds all 9 messages.

So in total: 2 phases × (√p − 1) steps each = 2(√p − 1) steps.

6. Cost analysis on different topologies

TopologyStepsCost
Ring (p nodes)p − 1(ts + m · tw) · (p − 1)
2-D Mesh (√p × √p)2(√p − 1)2 ts (√p − 1) + tw · m · (p − 1)
Hypercubelog pts · log p + tw · m · (p − 1)

7. Why is the m · tw · (p − 1) part the same in all three formulas?

Because no matter what shape we use, each node ultimately RECEIVES p − 1 different messages (one from every other node). That is a FIXED amount of data per node. Only the startup-cost component (ts) varies between topologies, because different shapes need different numbers of communication steps.

8. All-to-All Reduction (the dual)

  • Each node has p values; the j-th value is destined for node j.
  • After the operation, each destination node j receives the COMBINED value (sum / max / etc.) of the j-th values from all senders.
  • Implemented by reversing the all-to-all broadcast algorithm: forward instead of broadcast, but apply the reduction operator at the destination.
  • Cost is identical to all-to-all broadcast on the same topology.
  • MPI function: MPI_Reduce_scatter.

9. Applications

  • Matrix transpose operation (every row of A becomes a column of AT).
  • Fast Fourier Transform (FFT) algorithms.
  • Sample sort and bucket sort in parallel.
  • Parallel database join operations.
  • Distributed cache update — propagating local state to all peers.
  • n-body and molecular dynamics simulations.

Key points to remember

  • All-to-all broadcast: each node sends its OWN message to every other node.
  • Each node ends up with copies of ALL p messages.
  • Ring → p−1 steps; Mesh → 2(√p−1) steps using row+column phases; Hypercube → log p steps.
  • The transfer term m · tw · (p − 1) is the same in all topologies.
  • Dual operation is all-to-all reduction with same cost.
  • MPI: MPI_Allgather and MPI_Reduce_scatter.

Q3 — Scatter and Gather Operations

Asked 4 times(May'23, May'24, May'25, Nov'25 — usually 4–6 marks)
How this question appeared in past papers:

(May'23 & May'25, 6 marks) Explain scatter and gather communication operation with diagram.

(May'24 & Nov'25, 4–6 marks) Explain scatter and gather operation.

Answer
A pizza box arrives at the office. The boss (the root) cuts the pizza into 4 slices and gives one DIFFERENT slice to each of 4 colleagues — that is scatter. Later, each colleague writes a review on paper; the boss collects all 4 papers and staples them together in order — that is gather.

1. Scatter — definition

Scatter (also called personalised one-to-all communication) is a collective operation in which the root computer holds an array of p · k items, splits it into p equal-size pieces, and sends ONE different piece to each of the p processes (including itself).

Root cuts a big array into p equal chunks and gives a DIFFERENT chunk to each process.

2. Characteristics of Scatter

  • It is a one-to-many personalised pattern.
  • Each receiver gets a DIFFERENT chunk (this is what makes it different from broadcast, where everyone gets the SAME message).
  • On a hypercube, it is performed in log p steps; the message size HALVES at every step (because the root sends successively smaller pieces to each subcube).
  • Total amount of data moved through the network = m · (p − 1) words.
  • MPI function: MPI_Scatter.
  • If chunk sizes are unequal, use MPI_Scatterv.
  • The dual (reverse) operation is Gather.

3. Diagram — Scatter

BEFORE SCATTER:                    AFTER SCATTER:
Root P0: [A B C D]                  P0: [A]
P1: empty                           P1: [B]
P2: empty                           P2: [C]
P3: empty                           P3: [D]
                                   (each process got a DIFFERENT piece)

4. MPI function signature (good to write in answer)

MPI_Scatter(
    void* sendbuf,    // source buffer (only the root reads from this)
    int   sendcount,  // number of items sent to EACH process
    MPI_Datatype sendtype,
    void* recvbuf,    // every process writes its piece here
    int   recvcount,  // number of items each process receives
    MPI_Datatype recvtype,
    int   root,       // rank of the root process
    MPI_Comm comm     // communicator (e.g. MPI_COMM_WORLD)
);

5. Cost of Scatter (on hypercube)

Tscatter = ts · log p + tw · m · (p − 1)

6. Applications of Scatter

  • Distributing rows or blocks of a large matrix to worker processes for parallel matrix-vector multiplication.
  • Splitting a big input file into smaller chunks for parallel processing (MapReduce input split).
  • Distributing different mini-batches of training data to GPU workers in deep learning.
  • Assigning different sub-problems in parallel divide-and-conquer algorithms.
  • Distributing graph partitions to processors for parallel graph algorithms.

7. Gather — definition

Gather (also called personalised all-to-one communication) is the EXACT REVERSE of scatter: each process sends its k-item piece to the root, which concatenates them in process-rank order to form one large buffer of p · k items.

Each process sends its piece to root; root joins all pieces into one big array.

8. Characteristics of Gather

  • It is a many-to-one personalised pattern.
  • The result on the root is a concatenation of all senders' pieces in rank order (P0's piece first, then P1's, then P2's, …).
  • Implementation just runs scatter "in reverse"; cost is identical to scatter.
  • MPI function: MPI_Gather.
  • If the result should be available on EVERY process (not just root), use MPI_Allgather.
  • If chunks have unequal sizes, use MPI_Gatherv.

9. Diagram — Gather

BEFORE GATHER:                AFTER GATHER:
P0: [A]                        Root P0: [A B C D]
P1: [B]                        P1: empty
P2: [C]                        P2: empty
P3: [D]                        P3: empty

10. Cost of Gather

Tgather = ts · log p + tw · m · (p − 1)

11. Applications of Gather

  • Collecting partial sums or partial results from each worker back to the master.
  • Assembling the final result matrix in parallel matrix multiplication.
  • Joining sorted sub-arrays in the final stage of parallel merge sort.
  • Collecting log files / status messages from worker nodes.
  • Building a global histogram from local histograms computed on each node.

12. Scatter vs Gather vs Broadcast vs Reduction (don't get confused!)

OperationDirectionWhat each receiver getsMPI call
BroadcastRoot → allSAME identical messageMPI_Bcast
ScatterRoot → allDIFFERENT chunk per processMPI_Scatter
ReductionAll → rootCombined value (sum, max, …)MPI_Reduce
GatherAll → rootConcatenation of all chunksMPI_Gather
All-GatherAll → allConcatenated array on every nodeMPI_Allgather

Key points to remember

  • Scatter = root cuts a big array and gives a DIFFERENT piece to each process.
  • Gather = each process sends its piece to root; root concatenates them.
  • Both cost: ts · log p + tw · m · (p − 1) on hypercube.
  • Scatter is OPPOSITE of broadcast (different vs. same chunks).
  • Gather is OPPOSITE of reduction (concatenate vs. combine).
  • MPI: MPI_Scatter, MPI_Gather, MPI_Allgather.

Q4 — Circular Shift Operation

Asked 4 times(May'23, May'24, May'25, Nov'25 — usually 4–6 marks)
How this question appeared in past papers:

(May'23 & May'24, 4 marks) Explain circular shift operation.

(May'25, 5 marks) Write a short note on Circular shift on a mesh.

(Nov'25, 6 marks) Explain circular shift operation on mesh and hypercube network.

Answer
A musical-chairs party — when the music starts, every child passes their toy to the friend sitting on the right. If the music plays for 3 seconds, every toy moves 3 seats to the right. The fancy technical name for this rotation is circular shift, and the number of seats to move is the "shift distance" q.

1. Definition

Circular Shift is a permutation communication operation in which every node i sends its data to node (i + q) mod p, where q is the shift distance and p is the total number of nodes. So all data effectively rotates around the network by q positions.

Every node rotates its data by q positions around the network — like an array rotation.

2. Characteristics and properties

  • It is a one-to-one (permutation) operation — same number of words leave and arrive at each node.
  • Two main variants:
    • End-around (cyclic) shift — data that "falls off" one end re-enters from the other end.
    • Compaction shift — no wrap-around; nodes at the boundary lose or gain padding.
  • The shift direction can be either left or right; the implementation picks whichever needs fewer hops.
  • If q = 0 or q = p, the operation is a no-op (nothing changes).
  • It is heavily used in matrix algorithms (Cannon's matrix multiplication), image processing, and stencil computations.
  • On a hypercube, it uses e-cube routing dimension-by-dimension.

3. Simple example (4 nodes, shift by 1 to the right)

Before:   P0:[A]   P1:[B]   P2:[C]   P3:[D]
After:    P0:[D]   P1:[A]   P2:[B]   P3:[C]
                  (each item rotated 1 step to the right)

4. Circular shift on a Ring

Each node sends its data q hops along the chosen direction. Since on a ring you can go either left or right, we choose the shorter direction:

Tring = (ts + m · tw) · min(q, p − q)

Working:

  1. Compute d = min(q, p − q) and decide direction.
  2. Repeat d times:
    • Each node sends its current data to neighbour in chosen direction.
    • Each node receives data from neighbour in opposite direction.

5. Circular shift on a 2-D Mesh (√p × √p)

Done in 2 phases:

  1. Row shift: each row independently shifts its data by (q mod √p) positions along the row.
  2. Column compensation: rows whose data wrapped around during the row shift are shifted up by one row.
3 × 3 mesh, shift by q = 4 (= 1 row + 1 column):

   Before              After
   1 2 3               6 7 8
   4 5 6               9 1 2
   7 8 9               3 4 5
Tmesh ≈ 2 (ts + m · tw · √p / 2)

6. Circular shift on a Hypercube

Uses dimension-by-dimension routing called E-cube routing: each node forwards the data along the dimensions in which the destination ID differs from the current node's ID.

Done in O(log p) steps for any power-of-2 shift.

Thypercube ≈ (ts + m · tw) · log p

7. Cost comparison across topologies

TopologyCostBig-O
Linear array(ts + m · tw) · qO(p)
Ring(ts + m · tw) · min(q, p − q)O(p)
2-D Mesh≈ 2(ts + m · tw · √p / 2)O(√p)
Hypercube≈ (ts + m · tw) · log pO(log p)

8. Applications of Circular Shift

  • Cannon's matrix multiplication algorithm — A-blocks are shifted LEFT and B-blocks are shifted UP after each iteration.
  • Stencil computations in weather modelling, computational fluid dynamics, image processing.
  • Sorting on a mesh — odd-even merge sort, shear sort.
  • Image rotation on distributed-memory systems.
  • Distributed hash-table re-balancing when nodes are added or removed.
  • Convolution operations in distributed CNNs.

Key points to remember

  • Circular shift = every node i sends data to (i + q) mod p.
  • Ring cost = O(min(q, p − q)); Mesh = O(√p); Hypercube = O(log p).
  • Mesh uses 2 phases: row shift + column compensation.
  • Hypercube uses E-cube routing (dimension-by-dimension).
  • Used in Cannon's algorithm, stencil computations, and parallel sorting.

Q5 — Prefix-Sum (Scan) Operation

Asked 3 times(May'23, May'25, Nov'25 — usually 4–6 marks)
How this question appeared in past papers:

(May'23 & Nov'25, 4 marks) Write a short note on prefix-sum operation.

(May'25, 6 marks) Explain prefix-sum operation for an eight-node hypercube.

Answer
A queue of 8 customers at the bank, each holding a different amount of money. Each customer wants to know: "What is the TOTAL money held by ME and everyone in front of me?" Customer 1 knows only their own amount. Customer 2 knows their own + Customer 1's. Customer 3 knows the first three combined. This running cumulative total is the prefix sum.

1. Definition

Prefix-Sum (also called Scan or Parallel Prefix) is a collective operation in which every node i (holding value xi) computes the cumulative combination si = x0 ⊕ x1 ⊕ … ⊕ xi, where ⊕ is any associative operator (+, ×, max, min, AND, OR, XOR).

Each node ends up with the running total of all values from node 0 up to and including itself.

2. Characteristics

  • The operator must be associative; if also commutative, additional optimizations are possible.
  • Two variants:
    • Inclusive scan: si INCLUDES xi itself.
    • Exclusive scan: si EXCLUDES xi (useful for computing array offsets / starting positions).
  • Uses the same recursive-doubling idea as broadcast and reduction.
  • Completes in log p steps on a hypercube.
  • Total work = p − 1 operator applications (same as serial), but done in only log p TIME on parallel hardware.
  • MPI functions: MPI_Scan (inclusive) and MPI_Exscan (exclusive).

3. Worked example (inclusive prefix sum with addition)

4. Algorithm — Prefix-Sum on a Hypercube

Algorithm PREFIX_SUM_HYPERCUBE (d, my_id, my_value):
   s = my_value         // running prefix sum
   m = my_value         // value used for forwarding
   for i = 0 to d − 1:
       partner = my_id XOR 2^i
       send m to partner
       receive m_recv from partner
       s = s + m_recv                  // every node updates its own total
       if (my_id AND 2^i) != 0:
           m = m + m_recv              // only higher-numbered keeps it
   end for
   return s

5. Step-by-step on an 8-node hypercube (3 dimensions = 3 steps)

  • Step 0 (dimension 0, bit 0 differs): pairs exchange — (0,1), (2,3), (4,5), (6,7).
  • Step 1 (dimension 1, bit 1 differs): pairs exchange — (0,2), (1,3), (4,6), (5,7).
  • Step 2 (dimension 2, bit 2 differs): pairs exchange — (0,4), (1,5), (2,6), (3,7).

6. Cost

Tprefix-sum = (ts + m · tw) · log p

7. Applications of Prefix-Sum

  • Computing bucket starting offsets in parallel radix sort.
  • Parallel stream compaction on GPUs (removing unwanted items from an array).
  • Parallel histogram construction.
  • Polynomial evaluation using Horner's rule in parallel.
  • Load balancing — figuring out how much work to assign each processor based on cumulative load.
  • Parallel lexicographic ranking of strings or tuples.
  • Parallel list ranking and many graph algorithms.
  • Building summed-area tables in image processing.

Key points to remember

  • Prefix-sum: node i finally holds x0 ⊕ x1 ⊕ … ⊕ xi.
  • Variants: inclusive scan and exclusive scan.
  • Cost on hypercube = (ts + m · tw) · log p.
  • Operator must be associative (e.g., +, ×, max, min).
  • MPI calls: MPI_Scan and MPI_Exscan.
  • Used heavily in radix sort, stream compaction, and load balancing.

Q6 — Blocking vs Non-Blocking Communication in MPI

Asked 2 times(May'23, May'24 — usually 6 marks)
How this question appeared in past papers:

(May'23 & May'24, 6 marks) Explain in detail Blocking and Non-Blocking Communication Using MPI.

Answer
A blocking call is like phoning someone and waiting on the line until they pick up — you cannot do anything else while waiting. A non-blocking call is like sending a WhatsApp message: you send it and immediately go back to work, then check later if they replied.

1. Definition

Blocking communication is an MPI mode where the function call does NOT return to the program until the message has been fully sent or received (i.e., until the local buffer is safe to reuse). The CPU sits idle during this wait.

Non-blocking communication is an MPI mode where the function call returns IMMEDIATELY, regardless of whether the actual transfer has completed. The transfer happens in the background, and the programmer must later check completion using MPI_Wait or MPI_Test.

2. Blocking Communication functions

  • MPI_Send — standard blocking send.
  • MPI_Recv — blocking receive.
  • MPI_Ssend — blocking synchronous send (returns only when receiver has begun to receive).
  • MPI_Bsend — blocking buffered send (uses an explicit user-provided buffer).
  • MPI_Rsend — blocking ready-mode send (assumes receiver is already ready).
  • MPI_Sendrecv — combined blocking send and receive in one call.

3. Properties of Blocking

  • The sender's buffer is safe to reuse as soon as MPI_Send returns.
  • The receiver's buffer contains the received data as soon as MPI_Recv returns.
  • Execution order is straightforward — easy to reason about.
  • CPU sits idle during the wait — no useful work can be done concurrently.
  • If two processes block on each other's sends, a deadlock may occur.

4. Advantages of Blocking

  • Easy to program — programmer does not have to track request handles.
  • Once the call returns, the operation is guaranteed complete.
  • Easier debugging due to deterministic ordering.
  • Lower memory overhead (no request objects to maintain).

5. Disadvantages of Blocking

  • CPU is idle while waiting — wastes processor time.
  • High risk of deadlock if communication patterns are not carefully designed.
  • Cannot overlap communication with computation.
  • Generally lower performance for data-intensive applications.

6. Non-Blocking Communication functions

  • MPI_Isend — non-blocking standard send (the "I" stands for Immediate).
  • MPI_Irecv — non-blocking receive.
  • MPI_Issend, MPI_Ibsend, MPI_Irsend — non-blocking variants of the other modes.
  • MPI_Wait(request, status) — blocks until the specified non-blocking operation completes.
  • MPI_Test(request, flag, status) — non-blocking check whether the operation has finished.
  • MPI_Waitall, MPI_Waitany, MPI_Waitsome — wait variants for multiple requests.

7. Properties of Non-Blocking

  • Function calls return immediately — CPU can do other useful work in parallel.
  • Each non-blocking call returns a request handle that must be later completed.
  • The data buffer must NOT be modified until MPI_Wait/MPI_Test confirms completion.
  • Multiple non-blocking operations can be in flight at the same time.
  • Enables overlapping of computation and communication for performance.

8. Advantages of Non-Blocking

  • Computation and communication can overlap → improved throughput.
  • Dramatically reduces the risk of deadlock.
  • Allows more flexible and complex communication patterns.
  • Better utilization of network bandwidth.
  • Useful in producer–consumer pipelines and asynchronous algorithms.

9. Disadvantages of Non-Blocking

  • Code is more complex; programmer must manage request handles.
  • Must remember not to touch the buffer until MPI_Wait succeeds.
  • Debugging is harder due to asynchronous behaviour.
  • Slight per-call overhead due to extra bookkeeping.

10. Code examples

// Blocking send/receive
int data = 42;
MPI_Send(&data, 1, MPI_INT, dest, tag, MPI_COMM_WORLD);    // waits until safe
MPI_Recv(&data, 1, MPI_INT, src,  tag, MPI_COMM_WORLD, &status);

// Non-blocking send/receive
int data = 42;
MPI_Request req;
MPI_Status status;

MPI_Isend(&data, 1, MPI_INT, dest, tag, MPI_COMM_WORLD, &req);
// CPU does other useful work here ...
do_some_computation();
MPI_Wait(&req, &status);    // confirm send is complete

Key points to remember

  • Blocking = call waits until safe to reuse buffer; simple but wastes CPU.
  • Non-blocking = returns immediately with a request handle; must call MPI_Wait later.
  • Non-blocking enables overlap of computation and communication.
  • Non-blocking avoids most deadlock scenarios.
  • MPI provides several variants: standard, synchronous, buffered, ready-mode.

Q7 — Improving the Speed of Communication Operations

Asked 2 times(May'24, Nov'25 — usually 4 marks)
How this question appeared in past papers:

(May'24, 4 marks) Comment on "Improving the Speed of Communication Operations".

(Nov'25, 4 marks) How to improve speed of communication operation.

Answer

The total time for a communication has two main parts: startup time (ts) and per-word transfer time (tw). We can reduce communication time by attacking either or both of these. Below are the 7 standard techniques used.

1. Splitting messages and routing along multiple paths

  • A single large message uses only ONE network path at a time, leaving other links idle.
  • SPLIT the message into smaller portions and send each through a DIFFERENT path simultaneously.
  • This increases effective bandwidth utilization.
  • Especially effective in hypercubes and tori, where multiple disjoint paths exist between every pair of nodes.

2. Pipelining of long messages

  • A long message is divided into smaller segments.
  • The receiver starts CONSUMING segment 1 while the sender is STILL SENDING segment 2.
  • This hides the time gap between successive segments — successive stages of the transmission overlap.
  • Similar in spirit to instruction pipelining inside a CPU.

3. Aggregating (combining) short messages

  • Each message pays a fixed startup cost ts, regardless of its size.
  • Sending 100 tiny messages pays this cost 100 times.
  • COMBINE many small messages into one larger message; ts is paid only ONCE.
  • Trade-off: small extra latency for the messages that wait while being buffered.

4. Choosing the right topology and routing strategy

  • The same operation has very different costs on different topologies (e.g., O(p) on ring vs. O(log p) on hypercube).
  • Choose efficient routing strategies such as e-cube routing on hypercubes or dimension-order routing on meshes.
  • Use better forwarding strategies: store-and-forward, cut-through, or wormhole routing, depending on hardware capability.

5. Overlapping computation with communication

  • Use NON-BLOCKING MPI calls (MPI_Isend, MPI_Irecv) so the processor performs useful computation while a message is in transit.
  • Place the corresponding MPI_Wait as late as possible to maximize overlap.
  • Especially effective when the computation between send and wait is comparable to the communication time.

6. Topology-aware mapping of processes to processors

  • Place processes that communicate frequently on PHYSICALLY ADJACENT processors.
  • This reduces the number of hops per message and lowers latency.
  • MPI implementations can be tuned with rank-reorder hints for topology awareness.

7. Use collective operations instead of point-to-point loops

  • Hand-rolled loops of MPI_Send/MPI_Recv are usually slower than the corresponding collective call (MPI_Bcast, MPI_Reduce, etc.).
  • MPI collectives are heavily optimized for the underlying network topology by the library implementor.

Key points to remember

  • Combine many small messages into one to reduce startup cost.
  • Split big messages across multiple paths for parallel transmission.
  • Hide latency using pipelining and non-blocking calls.
  • Pick topology-friendly routing (e-cube, dimension-order).
  • Prefer MPI collectives over hand-rolled point-to-point loops.

Q8 — All-Reduce and Combined Broadcast/Reduce

Asked 1 time(May'25 — overview question, 6 marks)
How this question appeared in past papers:

(May'25, 6 marks) Explain Broadcast and Reduce operation with diagram.

Answer

1. Quick recap of Broadcast

One root sends an identical message to every other node. Implemented in MPI as MPI_Bcast. Done in log p steps using recursive doubling.

              Root P0
                │
      ┌─────┬───┼───┬─────┐
      ↓     ↓   ↓   ↓     ↓
      P1    P2  P3  P4    P5
       (every receiver gets the SAME copy)

2. Quick recap of Reduce

Every node contributes a value; the destination receives the combined result. Implemented in MPI as MPI_Reduce. Done in log p steps using recursive halving.

      P1    P2  P3  P4    P5
      │     │   │   │     │
      ↓     ↓   ↓   ↓     ↓
      └─────┴───⊕───┴─────┘
                │
             Root P0
      (root has combined value)

3. All-Reduce

  • An all-reduce is conceptually a Reduce followed by a Broadcast, so that EVERY node ends up with the combined result (not just the root).
  • It can be implemented more efficiently than separate reduce + broadcast by using a butterfly pattern on a hypercube in log p steps.
  • MPI function: MPI_Allreduce.
  • Heavily used in distributed deep-learning training — averaging gradient values across all GPU workers after each batch.

Key points to remember

  • Broadcast: one to all, same message.
  • Reduce: all to one, combined result.
  • All-Reduce: all to all, combined result on EVERY node.
  • All three complete in log p steps on a hypercube.
  • All-Reduce is the dominant communication primitive in distributed deep learning.

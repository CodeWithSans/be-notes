Preliminary Concepts (Read Once)

A normal CPU is like a few very smart, well-paid professors who can each handle complicated tasks one at a time. A GPU is like an army of cheap student volunteers — each one can only do simple work, but there are THOUSANDS of them, so they finish huge piles of simple jobs (like drawing pixels, multiplying matrices, training neural networks) very fast.

1. CPU vs GPU at a glance

AspectCPUGPU
Number of coresFew (4–16), very powerfulMany (thousands), simpler
Designed forLow latency on ONE taskHigh throughput on MANY tasks
Control logicComplex (branch prediction etc.)Simple
CacheLarge per coreSmall per core, many caches
Best forSequential, branchy codeData-parallel, repetitive code
MemoryFew GB to TB, slow accessFew to dozens of GB, very fast

2. Big difference: latency vs throughput

  • Latency = how fast ONE task completes. CPU wins here.
  • Throughput = how many tasks complete per second total. GPU wins here.
  • Example: A CPU does one tax form in 1 minute. A GPU does 1000 tax forms in 5 minutes. CPU is faster per form, but GPU finishes the whole pile faster.

3. What is CUDA?

CUDA = Compute Unified Device Architecture. It is NVIDIA's parallel computing framework — a way to write programs in a C-like language that run on NVIDIA GPUs.
  • Released by NVIDIA in 2006.
  • Only works on NVIDIA GPUs (AMD GPUs use OpenCL / ROCm instead).
  • Lets you write thousands of parallel threads as easily as writing a normal C function.

Q1 — Processing Flow of a CUDA-C Program

Asked 3 times(May'23, May'24, Nov'25 — usually 6 marks)
How this question appeared in past papers:

(May'23 & Nov'25, 6 marks) Describe processing flow of CUDA-C program with diagram.

(May'24, 6 marks) Describe processing flow of cuda along with cuda c functions.

Answer
Imagine a manager (CPU/host) outsourcing a heavy task to a factory (GPU/device). The manager: (1) prepares the raw material on the office desk, (2) calls a courier to copy it to the factory, (3) tells the factory workers what to make and how many copies, (4) waits while they work, (5) gets the finished products couriered back to the office. CUDA-C programs follow this exact flow.

1. Definition

The processing flow of a CUDA-C program is the sequence of steps a CUDA program performs to use the GPU: allocate memory on the GPU, copy input data to GPU, launch a kernel function that runs on many GPU threads in parallel, copy the result back to the CPU, and free the GPU memory.

2. The 5 standard steps in every CUDA-C program

  1. Allocate memory on the host (CPU) and initialise input data.
  2. Allocate memory on the device (GPU) using cudaMalloc.
  3. Copy input data from host RAM to device RAM using cudaMemcpy(..., cudaMemcpyHostToDevice).
  4. Launch the kernel on the GPU: myKernel<<<gridDim, blockDim>>>(args);
    Many threads run in parallel, each working on a small piece of data.
  5. Copy result back from device to host using cudaMemcpy(..., cudaMemcpyDeviceToHost), then free device memory with cudaFree.

5. Example CUDA-C program (vector addition skeleton)

#include <cuda_runtime.h>

__global__ void vecAdd(float* A, float* B, float* C, int N) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < N) C[i] = A[i] + B[i];
}

int main() {
    int N = 1000;
    float *h_A, *h_B, *h_C;             // host pointers
    float *d_A, *d_B, *d_C;             // device pointers

    // Step 1: allocate & init host data
    h_A = (float*)malloc(N*sizeof(float));
    // ... fill h_A and h_B with data ...

    // Step 2: allocate device memory
    cudaMalloc(&d_A, N*sizeof(float));
    cudaMalloc(&d_B, N*sizeof(float));
    cudaMalloc(&d_C, N*sizeof(float));

    // Step 3: copy host → device
    cudaMemcpy(d_A, h_A, N*sizeof(float), cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, N*sizeof(float), cudaMemcpyHostToDevice);

    // Step 4: launch kernel
    int threadsPerBlock = 256;
    int blocks = (N + threadsPerBlock - 1) / threadsPerBlock;
    vecAdd<<<blocks, threadsPerBlock>>>(d_A, d_B, d_C, N);

    // Step 5: copy device → host
    cudaMemcpy(h_C, d_C, N*sizeof(float), cudaMemcpyDeviceToHost);

    // Step 6: cleanup
    cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);
    free(h_A); free(h_B); free(h_C);
    return 0;
}

6. Direction codes for cudaMemcpy

  • cudaMemcpyHostToDevice — copy from CPU memory to GPU memory
  • cudaMemcpyDeviceToHost — copy from GPU back to CPU
  • cudaMemcpyDeviceToDevice — copy within GPU memory
  • cudaMemcpyHostToHost — copy within host (rarely used in CUDA code)

Key points to remember

  • 5-step flow: allocate host data → cudaMalloc → cudaMemcpy(H→D) → kernel launch → cudaMemcpy(D→H) → cudaFree.
  • Host = CPU side; Device = GPU side.
  • Key functions: cudaMalloc, cudaFree, cudaMemcpy, cudaDeviceSynchronize.
  • Kernel launch syntax: kernel<<<grid, block>>>(args).
  • Always copy data back to host before printing/using on CPU.

Q2 — CUDA Architecture in Detail

Asked 2 times(May'24, May'25 — usually 8–9 marks)
How this question appeared in past papers:

(May'24, 8 marks) Draw and explain CUDA architecture in details.

(May'25, 9 marks) Explain CUDA architecture in details.

Answer
A GPU is like a big company building. The building has many DEPARTMENTS (called Streaming Multiprocessors or SMs). Each department has many EMPLOYEES (CUDA cores). Each department has its own small meeting room (shared memory) and small lockers (registers). The whole company shares one big warehouse (global memory). The company gets work orders from the CEO (CPU) over a road (PCIe bus).

1. Definition

CUDA architecture is the hardware-and-software model used by NVIDIA GPUs to run thousands of parallel threads. It consists of multiple Streaming Multiprocessors (SMs), each containing many CUDA cores, plus a memory hierarchy and a software stack (driver, runtime, compiler).

2. Major hardware components

(a) Streaming Multiprocessor (SM)

  • The basic processing unit of a GPU. A high-end GPU has dozens of SMs (e.g., 80+ in A100).
  • Each SM contains:
    • Many CUDA cores (Stream Processors) — execute arithmetic operations.
    • Warp scheduler — schedules groups of 32 threads (called warps).
    • Special Function Units (SFUs) — handle sin, cos, sqrt, log, etc.
    • Registers — fastest memory, private per thread.
    • Shared memory — small, fast, on-chip; shared by all threads in a block.
    • L1 cache and instruction cache.

(b) Memory hierarchy

  • Registers (fastest, smallest, per-thread).
  • Shared memory (very fast, on-chip, per-block).
  • Constant memory (cached, read-only on device, set by host).
  • Texture memory (cached, optimized for 2D spatial reads).
  • Global memory (largest, slowest, off-chip; visible to all threads and host).
  • L2 cache — shared across all SMs.

(c) Host-Device interconnect

  • The CPU (host) talks to the GPU (device) over the PCIe bus.
  • Modern systems also support NVLink — faster GPU-to-GPU communication.
  • All data must be moved between host and device memory explicitly.

4. Software stack

  • CUDA Driver — low-level interface to GPU hardware.
  • CUDA Runtime API — high-level functions like cudaMalloc, cudaMemcpy.
  • NVCC compiler — compiles CUDA C/C++ source into machine code.
  • CUDA libraries — cuBLAS (linear algebra), cuFFT (FFT), cuDNN (deep neural nets), Thrust (parallel STL).

5. Execution model

  • The host launches a kernel → kernel runs on the device.
  • Each kernel launch creates a grid of blocks, each block contains many threads.
  • Threads are grouped into warps of 32; one warp executes the same instruction at the same time (SIMT model).
  • Blocks are assigned to SMs; an SM runs multiple blocks concurrently if resources allow.

Key points to remember

  • GPU = many SMs; each SM = many CUDA cores + shared memory + registers.
  • Memory hierarchy: registers → shared → constant → texture → global.
  • Host (CPU) and Device (GPU) connected via PCIe / NVLink.
  • Execution: kernel → grid → blocks → threads → warps (32 threads).
  • Software: NVCC compiler + CUDA Runtime API + libraries (cuBLAS, cuDNN).

Q3 — What is CUDA + Programming Languages + Applications

Asked 2 times(May'23, Nov'25 — usually 8 marks)
How this question appeared in past papers:

(May'23 & Nov'25, 8 marks) What is CUDA? Explain different programming languages support in CUDA. Discuss any three applications of CUDA.

Answer

1. What is CUDA?

CUDA (Compute Unified Device Architecture) is NVIDIA's parallel computing platform and programming model that lets developers use NVIDIA GPUs for general-purpose computation (not just graphics).
  • Introduced by NVIDIA in 2006.
  • Provides a software stack to write programs that run thousands of threads in parallel on GPU cores.
  • Includes: a compiler (NVCC), runtime libraries, debugging tools, and high-level libraries (cuBLAS, cuFFT, cuDNN, Thrust).
  • Programs are typically written in C/C++ with special __global__, __device__ qualifiers.

2. Programming languages supported by CUDA

  • CUDA C/C++ — the original and most popular language. Compiled by NVCC.
  • CUDA Fortran — provided by NVIDIA HPC SDK; popular in scientific computing.
  • CUDA Python — via libraries like:
    • Numba — JIT-compiles Python functions to CUDA kernels.
    • PyCUDA — Python wrapper around CUDA driver API.
    • CuPy — NumPy-like API that runs on GPU.
  • Java — via JCuda, Aparapi, JCublas.
  • .NET / C# — via ManagedCUDA, CUDAfy.NET.
  • MATLAB — Parallel Computing Toolbox supports CUDA arrays.
  • Julia — via CUDA.jl.
  • R — via gpuR and similar packages.
  • Higher-level frameworks: OpenACC (compiler-directive based), OpenCL (cross-vendor, conceptually similar).

3. Three (or more) important applications of CUDA

(a) Deep Learning

  • Training and inference of neural networks (CNNs, RNNs, transformers).
  • Frameworks: TensorFlow, PyTorch — they all use CUDA under the hood via cuDNN.
  • A GPU can train a CNN 10–100× faster than a CPU.

(b) Scientific Simulations

  • Molecular dynamics simulations (drug discovery).
  • Weather and climate forecasting.
  • Computational fluid dynamics (CFD).
  • Astrophysics simulations (n-body, galaxy formation).

(c) Image and Video Processing

  • Real-time filters, encoding/decoding (H.264, H.265).
  • Computer vision — object detection, face recognition.
  • Super-resolution and AI-based upscaling.
  • Medical imaging — CT/MRI reconstruction.

(d) Other major applications

  • Cryptography and password recovery (brute-force attacks).
  • Cryptocurrency mining (SHA-256, Ethash).
  • Financial modelling — Monte Carlo simulations, option pricing.
  • Bioinformatics — DNA sequence alignment, protein folding (AlphaFold).
  • Ray tracing and 3D rendering for films and games.
  • Database query acceleration (RAPIDS, BlazingSQL).

Key points to remember

  • CUDA = NVIDIA's framework to run parallel programs on GPU (since 2006).
  • Languages: C/C++, Fortran, Python (Numba/PyCUDA/CuPy), Java, .NET, MATLAB, Julia, R.
  • Top 3 applications: Deep learning, scientific simulation, image/video processing.
  • Other major uses: cryptography, finance, bioinformatics, graphics.

Q4 — CUDA Memory Model + Thread Hierarchy

Asked 2 times(May'23, May'24 — usually 6–8 marks)
How this question appeared in past papers:

(May'23, 8 marks) Explain CUDA memory model. Discuss thread hierarchy.

(May'24, 6 marks) Explain cuda memory model in brief.

Answer
Think of a CUDA program like a big school:
  • The whole SCHOOL = a Grid (one kernel launch).
  • Each CLASSROOM = a Block (group of threads that can share a board and notes).
  • Each STUDENT = a Thread (does its own little piece of work).
  • Each student has a PRIVATE NOTEBOOK (registers).
  • Classmates share a CLASSROOM BLACKBOARD (shared memory).
  • The whole school shares a LIBRARY (global memory) and a REFERENCE BOOK (constant memory).

Part A — Thread Hierarchy

1. The three levels

  • Grid — the topmost unit; one kernel launch creates one grid.
  • Block — a grid contains many blocks, arranged in 1-D, 2-D, or 3-D.
  • Thread — a block contains many threads, also in 1-D / 2-D / 3-D.

2. Built-in variables inside the kernel

  • threadIdx.x / .y / .z — index of this thread inside its block.
  • blockIdx.x / .y / .z — index of this block inside the grid.
  • blockDim.x / .y / .z — number of threads per block.
  • gridDim.x / .y / .z — number of blocks in the grid.

3. Global thread index (1-D)

global_id = blockIdx.x * blockDim.x + threadIdx.x

4. Warps

  • Threads inside a block are further grouped into warps of 32 threads.
  • All 32 threads in a warp execute the SAME instruction at the same time (SIMT — Single Instruction Multiple Threads).
  • If threads in a warp take different branches, performance drops (called warp divergence).

5. Diagram of thread hierarchy

   ┌─────────────────────  Grid  ─────────────────────┐
   │  Block(0,0)   Block(1,0)   Block(2,0)            │
   │  Block(0,1)   Block(1,1)   Block(2,1)            │
   │       ...        ...           ...                │
   └──────┬────────────────────────────────────────────┘
          │
          ▼
   ┌─────────  Block(i,j)  ─────────┐
   │   Thread(0,0) Thread(1,0) ...  │
   │   Thread(0,1) Thread(1,1) ...  │
   │     ...        ...   ...        │
   └────────────────────────────────┘
        ▲
        │
   Each warp = 32 consecutive threads

Part B — CUDA Memory Model

6. Types of memory in CUDA

MemoryLocationScopeSpeedLifetime
RegistersOn-chipOne threadFastestThread
Local memoryOff-chip (in global)One threadSlowThread
Shared memoryOn-chipAll threads in 1 blockVery fastBlock
Global memoryOff-chip DRAMAll threads, hostSlowApplication
Constant memoryOff-chip, cachedAll threads (read-only on device)Fast (cached)Application
Texture memoryOff-chip, cachedAll threads, 2D-optimizedFast (cached)Application

7. Rules to remember

  • Threads in the same BLOCK can share data via shared memory and synchronize via __syncthreads().
  • Threads in DIFFERENT blocks can only share data via slow global memory.
  • The smaller the memory, the faster the access AND the more limited the scope.
  • Registers and shared memory are precious — using too many limits how many blocks can run on each SM.

8. Diagram of CUDA memory hierarchy

Per-thread:    [Registers]   [Local memory]
                   ↓               ↓
Per-block:     [Shared memory]
                   ↓
Per-grid:      [Global memory]  [Constant memory]  [Texture memory]
                                    ↑
                              Host RAM ← cudaMemcpy

Key points to remember

  • Thread hierarchy: Grid → Block → Thread. Warp = 32 threads.
  • Built-in vars: threadIdx, blockIdx, blockDim, gridDim.
  • Global thread ID (1D): blockIdx.x · blockDim.x + threadIdx.x.
  • Memory types: registers (per thread), shared (per block), global (per app), constant, texture.
  • Use __syncthreads() to synchronize within a block.
  • Blocks cannot directly sync with each other inside a single kernel launch.

Q5 — Terms in CUDA: Device, Host, Device Code, Kernel

Asked 2 times(May'23, Nov'25 — usually 4 marks)
How this question appeared in past papers:

(May'23 & Nov'25, 4 marks) Explain the following terms in CUDA: device, host, device code, Kernel.

Answer

1. Host

  • The CPU and its system RAM.
  • Runs the main C/C++ program.
  • Responsible for: allocating GPU memory, transferring data, launching kernels, collecting results.
  • "Host code" = code that runs on the CPU.

2. Device

  • The GPU and its memory.
  • Acts as a co-processor to the host.
  • Executes parallel kernels launched by the host.
  • Cannot start work on its own — only does what host tells it.

3. Device Code

  • The functions that run on the GPU.
  • Marked with one of these qualifiers:
    • __global__ — kernel; callable from host, runs on device.
    • __device__ — helper function; callable from device, runs on device.
    • __host__ — runs on host (default for normal C functions).
  • Compiled by NVCC into GPU machine code (SASS / PTX).

4. Kernel

  • A special function marked with __global__.
  • Called from the host, but RUNS on the device.
  • Always returns void.
  • Executed in parallel by many threads.
  • Launched using the triple-angle-bracket syntax: kernel<<<gridDim, blockDim>>>(args);

5. Summary table

TermRefers toRuns onCalled by
HostCPU + RAM——
DeviceGPU + GPU memory——
Host codeNormal C/C++ codeHost (CPU)Anywhere
Device codeCode with __device__ or __global__Device (GPU)Host or device
KernelFunction with __global__Device (GPU)Host only

Key points to remember

  • Host = CPU side. Device = GPU side.
  • Device code runs on GPU; marked with __global__ or __device__.
  • Kernel = __global__ function, called from host, runs on device, returns void.
  • Kernel launch syntax: kernel<<<grid, block>>>(args).

Q6 — Block Dimension, Grid Dimension, and Vector Addition Kernel

Asked 1 time(May'23 — 6 marks)
How this question appeared in past papers:

(May'23, 6 marks) What is block dimension and grid dimension in CUDA? Write a CUDA kernel for addition of two vectors and explain how it will calculate addition using threads.

Answer

1. Block dimension and Grid dimension

  • blockDim — number of threads in one block, in each dimension (x, y, z). A block can be 1-D, 2-D, or 3-D.
  • gridDim — number of blocks in the grid, in each dimension. A grid can be 1-D, 2-D, or 3-D.
  • blockIdx — the index of THIS block within the grid.
  • threadIdx — the index of THIS thread within its block.
  • Specified in the kernel launch: kernel<<<gridDim, blockDim>>>(args).
  • Total threads created = gridDim.x · blockDim.x (for 1-D launch).

2. Why split into blocks and threads?

  • Blocks let the GPU schedule work across many SMs.
  • Threads inside a block can communicate via shared memory and sync with __syncthreads().
  • Maximum threads per block is usually 1024.

3. CUDA kernel for vector addition

__global__ void vectorAdd(float* A, float* B, float* C, int N) {
    // compute global index of this thread
    int i = blockIdx.x * blockDim.x + threadIdx.x;

    // bounds check — important when N is not a multiple of blockDim
    if (i < N) {
        C[i] = A[i] + B[i];
    }
}

4. How it calculates addition using threads (step-by-step)

  1. The host calls: vectorAdd<<<(N+255)/256, 256>>>(d_A, d_B, d_C, N);
  2. This creates a grid of (N+255)/256 blocks, each with 256 threads → total ≥ N threads.
  3. Each thread computes its global index i = blockIdx.x * 256 + threadIdx.x.
  4. Thread i reads A[i] and B[i] from global memory.
  5. It adds them and writes the result to C[i].
  6. All threads do this IN PARALLEL — instead of N sequential additions, the GPU does them all almost at once.

5. Example trace (N = 1000, blockDim = 256)

  • gridDim = ceil(1000/256) = 4 blocks.
  • Total threads = 4 × 256 = 1024 (24 extra threads beyond N).
  • Thread 0 in block 0: i = 0 → C[0] = A[0] + B[0].
  • Thread 5 in block 1: i = 1·256 + 5 = 261 → C[261] = A[261] + B[261].
  • Threads with i ≥ 1000 do nothing (caught by the if (i < N) check).

Key points to remember

  • blockDim = threads per block; gridDim = blocks in grid.
  • Total threads = gridDim · blockDim.
  • Global thread index (1D) = blockIdx.x · blockDim.x + threadIdx.x.
  • Always include bounds check if (i < N) when array size isn't a multiple of blockDim.
  • Each thread does one element of the addition — true parallel execution.

Q7 — Kernel and Kernel Launch Arguments

Asked 1 time(May'23 — 4 marks)
How this question appeared in past papers:

(May'23, 4 marks) What is a Kernel in CUDA? What is kernel launch? Explain arguments that can be specified in a Kernel launch.

Answer

1. What is a Kernel?

  • A kernel is a special function declared with __global__.
  • It is called from host code but executes on the device (GPU).
  • It is run in parallel by many threads.
  • It always returns void.
  • Example: 
    __global__ void myKernel(float* data, int N) { ... }

2. What is Kernel Launch?

  • Kernel launch = the act of telling the GPU to start executing a kernel.
  • Done from host code using the triple-angle-bracket syntax: 
    myKernel<<<gridDim, blockDim, sharedMem, stream>>>(args);
  • The launch returns IMMEDIATELY (it is asynchronous) — the CPU can do other work while the kernel runs.

3. Kernel launch arguments — the four inside <<< >>>

  1. gridDim (first argument, required)
    • Number of blocks in the grid.
    • Can be a single integer (1-D grid) or dim3 for 2-D / 3-D.
  2. blockDim (second argument, required)
    • Number of threads per block.
    • Maximum usually 1024 threads per block.
  3. sharedMem (third argument, optional)
    • Bytes of dynamically-allocated shared memory per block.
    • Used when shared memory size isn't known at compile time.
    • Default = 0.
  4. stream (fourth argument, optional)
    • The CUDA stream in which the kernel runs.
    • Streams allow concurrent kernel execution.
    • Default = stream 0 (the default stream).

4. Examples of kernel launches

// Simple 1-D launch
myKernel<<<100, 256>>>(data, N);

// 2-D grid and blocks (using dim3)
dim3 grid(8, 8);
dim3 block(16, 16);
myKernel2D<<<grid, block>>>(image, W, H);

// With dynamic shared memory (1024 bytes)
myKernel<<<100, 256, 1024>>>(data, N);

// With a non-default stream
cudaStream_t s;
cudaStreamCreate(&s);
myKernel<<<100, 256, 0, s>>>(data, N);

Key points to remember

  • Kernel = __global__ function, runs on GPU, called from host, returns void.
  • Kernel launch syntax: kernel<<<grid, block, sharedMem, stream>>>(args).
  • 4 launch arguments: gridDim, blockDim, sharedMem (opt), stream (opt).
  • Kernel launch is asynchronous — host doesn't wait.
  • Use cudaDeviceSynchronize() if host needs to wait for completion.

Q8 — CUDA C Program Execution at the Kernel Level

Asked 1 time(May'24 — 8 marks)
How this question appeared in past papers:

(May'24, 8 marks) Explain how the CUDA C program executes at the kernel level with example.

Answer

1. What "kernel-level execution" means

  • When the host calls kernel<<<grid, block>>>(args), the GPU spawns thousands of threads.
  • All threads run the SAME code (the kernel function), but each one works on a DIFFERENT piece of data based on its thread ID.
  • This is the SIMT (Single Instruction Multiple Threads) model.

2. Step-by-step execution flow

  1. Host launches kernel: CUDA runtime sends the launch request to the GPU.
  2. GPU's GigaThread Engine distributes blocks to available Streaming Multiprocessors (SMs).
  3. Each SM receives one or more blocks and divides each block into warps of 32 threads.
  4. The warp scheduler selects ready warps and issues instructions.
  5. All 32 threads in a warp execute the SAME instruction on different data.
  6. Each thread computes its global ID (blockIdx · blockDim + threadIdx) and uses it to access its data.
  7. Threads read/write global memory; threads in the same block can share data via shared memory.
  8. Threads in the same block sync at __syncthreads() barriers.
  9. When all threads in a block finish, the block exits; SM picks up the next block.
  10. When the entire grid finishes, control returns to the host.

3. Example — vector addition (kernel-level trace)

__global__ void vecAdd(float* A, float* B, float* C, int N) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < N) C[i] = A[i] + B[i];
}

// Host launch
vecAdd<<<4, 256>>>(d_A, d_B, d_C, 1000);

Kernel-level execution:

  • 4 blocks created: B0, B1, B2, B3, each with 256 threads.
  • Suppose GPU has 2 SMs. GigaThread sends B0 + B1 to SM0; B2 + B3 to SM1.
  • Each SM splits each block into 8 warps (256 ÷ 32 = 8).
  • Inside one warp (say warp 0 of block 0):
    • All 32 threads compute their global IDs (0, 1, 2, ..., 31).
    • All 32 threads read A[i] and B[i] from global memory (these can be coalesced into one transaction).
    • All 32 threads add and write to C[i].
  • Once all 8 warps of a block finish, the block is done. SM picks the next block.
  • When all 4 blocks finish, the kernel ends and control returns to host.

4. Important kernel-level concepts

  • SIMT: all threads in a warp execute the same instruction in lockstep.
  • Warp divergence: if threads in a warp take different branches, the warp executes both branches sequentially → slower.
  • Memory coalescing: when 32 threads in a warp access 32 consecutive memory locations, the GPU combines them into one transaction → faster.
  • Latency hiding: when one warp is waiting for memory, the SM schedules another warp to run — keeps GPU busy.

Key points to remember

  • Kernel runs in parallel; each thread has a unique ID.
  • Blocks distributed to SMs; each SM runs warps of 32 threads.
  • SIMT model: same instruction, different data.
  • Warp divergence and memory access patterns greatly affect performance.
  • Host can wait for kernel via cudaDeviceSynchronize().

Q9 — Advantages and Limitations of CUDA

Asked 1 time(May'24 — 4 marks)
How this question appeared in past papers:

(May'24, 4 marks) Write advantages and limitations of CUDA.

Answer

1. Advantages of CUDA

  • Massive parallelism — thousands of threads run simultaneously, ideal for data-parallel problems.
  • C-like syntax — easy learning curve for programmers familiar with C/C++.
  • Rich library ecosystem — cuBLAS, cuFFT, cuDNN, Thrust, RAPIDS provide tuned implementations.
  • Mature tools — Nsight debugger, profiler, and IDE integrations.
  • High performance — gives huge speedups for matrix operations, deep learning, image processing.
  • Wide adoption — used by all major AI frameworks (PyTorch, TensorFlow).
  • Direct hardware access — fine-grained control over memory, threads, streams.
  • Asynchronous execution — overlap computation and data transfer using streams.
  • Backward compatibility — programs written years ago still run on modern GPUs.

2. Limitations of CUDA

  • Vendor-locked — only runs on NVIDIA GPUs (AMD / Intel GPUs need OpenCL or SYCL).
  • PCIe transfer overhead — copying data between host and device can dominate small jobs.
  • Bad for branchy / irregular code — causes warp divergence; performance drops.
  • Limited GPU memory compared to host RAM (e.g., a GPU has 16–80 GB vs server with TBs of RAM).
  • Learning curve — must think in parallel, manage memory hierarchy, sync threads.
  • Debugging is harder than sequential CPU code.
  • Power consumption — high-end GPUs consume 300–700 W; expensive to cool and power.
  • Cost — top GPUs are expensive (₹ lakhs per card).
  • No automatic parallelization — programmer must manually identify parallelism.

  • Limitations: NVIDIA-only, PCIe overhead, warp divergence, limited memory, high cost.

Q11 — Managing GPU Memory

Asked 1 time(May'25 — 9 marks)
How this question appeared in past papers:

(May'25, 9 marks) Write a short note on: Managing GPU Memory.

Answer

1. Why GPU memory management matters

  • GPU memory is SEPARATE from CPU memory.
  • All data the kernel uses must first be COPIED to GPU memory.
  • GPU memory is LIMITED (typically a few to dozens of GB).
  • Memory transfers over PCIe are SLOW — minimize them.
  • Poor memory management = poor performance, even with fast kernels.

2. Types of GPU memory and how to manage each

Global memory

  • Allocated using cudaMalloc(&ptr, size).
  • Freed using cudaFree(ptr).
  • Largest but slowest.
  • Visible to all threads and the host.

Shared memory

  • Declared inside kernel with __shared__: 
    __shared__ float buf[256];
  • Very fast, small (~48–96 KB per SM).
  • Shared among threads of the SAME block.
  • Used for inter-thread data sharing and caching.

Constant memory

  • Declared with __constant__: 
    __constant__ float coeff[64];
  • Read-only on device, set by host using cudaMemcpyToSymbol.
  • Cached → fast access when all threads read the same value.

Texture memory

  • Read-only, optimized for 2D spatial access patterns.
  • Useful for image processing where neighbouring threads access neighbouring pixels.

Registers and local memory

  • Each thread has private registers (very fast, ~few KB).
  • If a thread uses too many variables, they overflow to "local memory" (which is actually slow off-chip global memory).

3. Memory management functions

FunctionPurpose
cudaMallocAllocate device memory
cudaFreeFree device memory
cudaMemcpyCopy between host and device
cudaMemcpyAsyncAsynchronous copy (overlaps with kernel)
cudaMemsetInitialize device memory to a value
cudaMallocHostAllocate pinned (page-locked) host memory for faster transfers
cudaMallocManagedAllocate unified memory accessible from both host and device
cudaMemcpyToSymbolCopy data into constant memory

4. Best practices for GPU memory management

  • Minimize host-device transfers — they are the slowest operation.
  • Reuse device memory instead of repeatedly allocating and freeing.
  • Use pinned (page-locked) host memory for faster transfers (cudaMallocHost).
  • Use unified memory (cudaMallocManaged) for simpler programming, at slight performance cost.
  • Coalesce global memory accesses — consecutive threads should access consecutive memory locations.
  • Use shared memory as a software-managed cache for data reused within a block.
  • Use constant memory for read-only data accessed by all threads (e.g., filter coefficients).
  • Free unused memory promptly using cudaFree.
  • Use asynchronous copies with streams to overlap data transfer and computation.

5. Common errors to avoid

  • Accessing device memory from host directly (dereferencing a device pointer) → segfault.
  • Forgetting to cudaFree → memory leak.
  • Using too much shared memory → kernel fails to launch.
  • Writing to constant memory from device → compile error.

Key points to remember

  • GPU memory types: global, shared, constant, texture, registers, local.
  • Use cudaMalloc/cudaFree/cudaMemcpy for global memory.
  • Minimize host-device transfers; use pinned/unified memory if helpful.
  • Coalesce accesses, use shared memory as cache.
  • Reuse buffers; free what you don't need.

Preliminary Concepts (Read Once)

This unit is about real-world applications of parallel computing — how classic algorithms (search, sort) are made parallel, and how distributed-computing frameworks like Kubernetes and GPU-based deep learning run massive workloads. It is the highest-scoring unit by question count in past papers.

Scope of Parallel Computing — where do we use it?

  • Scientific computing — weather, climate, molecular dynamics, astrophysics.
  • AI / ML — training huge neural networks, running inference at scale.
  • Big-data analytics — log analysis, search indexing, recommendation systems.
  • Computer graphics / vision — ray-tracing, real-time rendering, autonomous driving.
  • Cryptography — password cracking, blockchain mining.
  • Engineering simulations — finite element analysis, computational fluid dynamics.
  • Medical imaging — CT/MRI reconstruction, genome sequencing.
  • Real-time systems — autonomous driving, robotics, IoT data processing.

Why parallel computing is essential today

  • Single-CPU clock speed has plateaued (~3–4 GHz due to heat limits).
  • Data sizes are exploding (TB → PB → EB scale).
  • Modern hardware = multi-core CPUs, GPUs, clusters — must be exploited.

Q1 — Parallel Depth-First Search (DFS)

Asked 4 times(May'23, May'24, May'25, Nov'25 — usually 4–9 marks)
How this question appeared in past papers:

(May'23, 6 marks) Explain Parallel Depth First Search algorithm in detail.

(May'24, 6 marks) Explain Parallel Depth First Search algorithm in detail.

(May'25, 9 marks) Modify DFS for parallel execution and analyze its complexity.

(Nov'25, 4 marks) Explain Parallel Depth First Search algorithm in detail.

Answer
Imagine searching for a missing person in a huge forest. ONE search party (sequential DFS) walks deep along one trail, hits a dead end, backtracks, tries another trail. Slow! Many search parties (parallel DFS) split up — each party takes a different sub-area. When one party finishes its area, it asks another (still-busy) party to share some of its remaining unsearched trails. They keep working until everyone is simultaneously idle (= search complete).

1. Sequential DFS recap

  • Use a stack; start from a root node.
  • Explore deeply along one branch until a dead end.
  • Backtrack and try the next branch.
  • Complexity: O(V + E) where V = vertices, E = edges.

2. Why parallelize DFS?

  • State-space search (puzzles, satisfiability, game-tree search) has HUGE trees — billions of nodes.
  • Different processors can explore DIFFERENT subtrees simultaneously.
  • Speedup can be near-linear if the work is well-balanced.

3. The challenge: search tree shape is unknown

  • We cannot statically divide the tree — some subtrees are huge, others tiny.
  • Static splitting → some processors finish quickly and sit idle, others struggle with huge subtrees.
  • Solution: dynamic load balancing.

4. Parallel DFS algorithm with dynamic load balancing

  1. Initially, processor 0 starts with the root node on its stack; all other processors are idle.
  2. When a processor's stack becomes empty, it sends a "work request" to another (busy) processor (called the "donor").
  3. The donor splits its remaining stack into two halves and sends one half to the requester.
  4. Both processors continue DFS on their own portions.
  5. The search ends when all processors are simultaneously idle (need a termination-detection algorithm).

5. Work-splitting strategies

  • Stack splitting: donor splits its stack in half (top or alternate entries).
  • Half-split with cutoff: send only entries above a certain depth threshold (they have bigger subtrees).
  • Recursive halving: repeatedly split the stack as more idle processors request work.

6. Strategies for selecting a "donor" processor

  • Random polling — requester picks a random processor and asks.
  • Asynchronous round-robin — each requester maintains a pointer; asks next processor in order.
  • Global round-robin — a central counter decides the next donor.

7. Termination detection (the tricky part)

  • How do we know all processors are really idle and no new work is in transit?
  • Dijkstra's token termination algorithm: a token circulates among processors. If it returns "white" after a full round with no new work generated, the search terminates.
  • Other methods: spanning-tree-based detection, four-counter algorithm.

8. Complexity

  • Sequential DFS: O(V + E).
  • Parallel DFS with p processors: ideally O((V + E)/p), but actual depends on load balancing efficiency.
  • Communication overhead grows with number of work-stealing requests.
  • Best speedup when search tree is huge and reasonably balanced.

9. Applications of Parallel DFS

  • AI search problems (puzzles like 8-puzzle, n-queens, Sudoku).
  • SAT solvers (satisfiability problems).
  • Game-tree search (chess, Go).
  • Graph traversal in distributed systems.
  • Topological sort in parallel.
  • Pathfinding in robotics.

Key points to remember

  • Sequential DFS = O(V + E); Parallel DFS ≈ O((V + E)/p) with good load balancing.
  • Use dynamic load balancing — static splitting fails because tree is unknown.
  • Work-stealing: idle processor requests work from a busy "donor".
  • Donor splits its stack (stack splitting / half-split with cutoff).
  • Termination detection via Dijkstra's token algorithm.
  • Used in AI search, SAT solvers, game-tree search, graph algorithms.

Q2 — Parallel Breadth-First Search (BFS)

Asked 4 times(May'23, May'24, May'25, Nov'25 — usually 4–6 marks)
How this question appeared in past papers:

(May'23, 4 marks) Explain parallel BFS algorithm in brief.

(May'24, 6 marks) Explain BFS for parallel execution & analyze its complexity.

(May'25, 4 marks) Write a short note on Parallel BFS.

(Nov'25, 6 marks) Explain communication strategies for parallel BFS.

Answer
Imagine emergency workers searching a flooded building floor-by-floor. They first check the entrance (level 0), then all rooms reachable in 1 step (level 1), then all reachable in 2 steps (level 2), and so on. Parallel BFS lets MANY teams check different rooms of the SAME LEVEL at the same time, then everyone synchronizes before moving to the next level.

1. Sequential BFS recap

  • Use a queue; start from a source node.
  • Visit all nodes at distance 1 first, then distance 2, etc. (level-order traversal).
  • Complexity: O(V + E).

2. Parallel idea — Level-Synchronous BFS

  • All processors work TOGETHER on ONE level (frontier) at a time.
  • At level k, the frontier = set of nodes just discovered.
  • The frontier is divided among processors — each processor explores its share of nodes.
  • Their unvisited neighbours form the NEXT frontier.
  • A barrier synchronization ensures all processors finish level k before moving to level k+1.

3. Algorithm sketch

Frontier F = {source}
level = 0
while F is not empty:
    parallel for each v in F:
        for each neighbour u of v:
            if u not visited:
                mark u visited
                add u to NewFrontier
                dist[u] = level + 1
    barrier_sync_all_processors()
    F = NewFrontier
    NewFrontier = empty
    level = level + 1

4. Complexity

  • Sequential BFS: O(V + E).
  • Parallel BFS: O((V + E)/p + diameter · log p), approximately.
  • The diameter · log p term comes from synchronization between levels.
  • Best speedup on graphs with small diameter (well-connected).

5. Challenges in parallel BFS

  • Load imbalance — some processors may get bigger sub-frontiers than others.
  • Duplicate work — two processors may discover the same node and try to add it.
  • Synchronization cost — barrier after every level can be expensive.
  • Communication overhead — newly-discovered nodes must be sent to the right processor.

6. Applications

  • Shortest-path finding in unweighted graphs.
  • Social network analysis (degrees of separation).
  • Web crawling.
  • Network broadcasting.
  • Topological analysis of road / electrical networks.
  • Iterative deepening in AI search.

Key points to remember

  • Sequential BFS = O(V + E); Parallel BFS ≈ O((V + E)/p + diameter · log p).
  • Level-synchronous: all processors work on one frontier, then barrier-sync before next level.
  • Challenges: load imbalance, duplicate detection, synchronization, communication.
  • Best on small-diameter (well-connected) graphs.
  • Used in shortest paths, social networks, web crawling.

Q3 — Parallel Merge Sort

Asked 3 times(May'23, May'24, Nov'25 — usually 4–8 marks)
How this question appeared in past papers:

(May'23, 8 marks) Write short notes on: i) Parallel Merge sort ii) GPU applications.

(May'24, 8 marks) Compare an algorithm for sequential and parallel Merge sort. Analyze the complexity for the same.

(Nov'25, 8 marks) Write short notes on: i) Parallel Merge Sort ii) GPU Applications.

Answer
Imagine sorting 1000 exam papers by roll number. ONE teacher (sequential merge sort) divides the pile in half, sorts each half, then merges them — slow. p TEACHERS (parallel merge sort) each take 1000/p papers, sort their own pile independently, then pair up and merge their sorted piles, then those pairs pair up, until all 1000 are merged into one sorted pile.

1. Sequential Merge Sort recap

  • Divide-and-conquer algorithm.
  • Split array into two halves, recursively sort each half, then merge them.
  • Complexity: O(n log n).
  • Stable and not in-place.

2. Parallel Merge Sort — the idea

  1. Divide: distribute the unsorted array among p processors. Each gets n/p elements.
  2. Local sort: each processor sorts its own n/p elements using sequential merge sort. Time = O((n/p) log(n/p)).
  3. Pairwise merge: in log p rounds, pairs of processors merge their sorted runs. Active processors halve each round.
  4. Final result: after log p rounds, one processor holds the fully sorted array.

3. Algorithm pseudo-code

Algorithm PARALLEL_MERGE_SORT (A, p):
   // distribute A in chunks of n/p to each processor
   each processor i locally sorts its chunk → produces sorted run R[i]

   active_set = {0, 1, ..., p-1}
   for round = 1 to log p:
       for each pair (i, j) in active_set:
           processor i merges R[i] and R[j] into new R[i]
           processor j becomes inactive
       active_set = active_set \ inactive
   end
   // Now R[0] contains the fully sorted array

4. Complexity analysis

  • Local sort: O((n/p) log(n/p)).
  • Merge rounds: log p rounds; each round merges sorted runs of growing size.
  • Round r: each merge takes O(2^(r-1) · n/p) time.
  • Total merge time = O((n/p) · log p).
  • Total parallel time: Tp = O((n/p) log n).
  • Speedup: ≈ p (when n ≫ p).
  • Cost = p · Tp = O(n log n) — same as sequential, so cost-optimal.

5. Comparison: Sequential vs Parallel Merge Sort

AspectSequentialParallel
Processors used1p
Local sortn log n(n/p) log (n/p)
Merge phaselog n levels, sequentiallog p levels, parallel pairs
Total timeO(n log n)O((n/p) log n)
Speedup—Up to O(p)
CommunicationNoneO(n) per merge round
MemoryO(n)O(n) total, O(n/p) per processor

6. Variants

  • Bitonic merge sort — uses a fixed network of comparators; ideal for hardware.
  • Sample sort — sample pivot elements first to ensure balanced partitions.
  • Parallel multi-way merge — instead of pairing, merge multiple runs at once.

Key points to remember

  • Sequential: O(n log n); Parallel: O((n/p) log n).
  • 3 phases: distribute → local sort → pairwise merge (log p rounds).
  • Cost-optimal: total cost = O(n log n) = sequential cost.
  • Speedup ≈ p when n ≫ p.
  • Stable, requires O(n) extra space.

Q4 — GPU Applications

Asked 3 times(May'23, May'24, Nov'25 — usually 4–8 marks)
How this question appeared in past papers:

(May'23 & Nov'25, 8 marks) Write short notes on: ii) GPU applications.

(May'24, 4 marks) Write short note on GPU Applications.

Answer

GPUs are no longer used only for graphics — they are now general-purpose accelerators for any task that can be broken into thousands of similar small computations. This is called GPGPU (General-Purpose computing on GPU).

1. Major application areas

(a) Deep Learning

  • Training and inference of neural networks (CNNs, RNNs, Transformers).
  • Frameworks: TensorFlow, PyTorch — built on CUDA / cuDNN.
  • Models like GPT, BERT, ResNet are trained on GPU clusters.
  • 10–100× faster than CPU training.

(b) Scientific Simulations

  • Molecular dynamics, drug discovery.
  • Climate and weather forecasting.
  • Computational fluid dynamics (CFD).
  • Astrophysical simulations (n-body, galaxy formation).

(c) Image and Video Processing

  • Real-time filters, color correction.
  • Video encoding/decoding (H.264, H.265, AV1).
  • Super-resolution and AI upscaling.
  • Medical imaging — CT/MRI reconstruction.

(d) Computer Vision

  • Object detection (YOLO, SSD).
  • Face recognition, license-plate recognition.
  • Autonomous driving perception (lane detection, obstacle detection).

(e) Cryptography and Mining

  • Password cracking (brute-force, dictionary).
  • Hashing (SHA-256 for Bitcoin, Ethash for Ethereum).
  • Encryption acceleration.

(f) Financial Modelling

  • Monte Carlo simulations for option pricing.
  • Risk analysis (value-at-risk computations).
  • High-frequency trading algorithms.

(g) Bioinformatics

  • DNA sequence alignment (BLAST on GPU).
  • Protein folding simulations (AlphaFold).
  • Genomic data analysis.

(h) Ray-Tracing and 3D Rendering

  • Real-time rendering in games (NVIDIA RTX, hardware ray-tracing).
  • Offline rendering for movies (Pixar, Disney).
  • Virtual reality (VR) and augmented reality (AR).

(i) Database query acceleration

  • GPU-accelerated SQL (BlazingSQL, RAPIDS cuDF).
  • Real-time analytics on huge datasets.

(j) Engineering simulations

  • Finite Element Analysis (FEA) for structural mechanics.
  • Crash simulations for automotive design.
  • Electromagnetic field simulation.

2. Why GPUs win in these applications

  • Thousands of CUDA cores → ideal for data-parallel workloads.
  • Very high memory bandwidth (GDDR6 / HBM2).
  • Mature software stack (CUDA, cuDNN, cuBLAS, TensorRT).
  • Hardware specifically designed for SIMD/SIMT execution.

Key points to remember

  • Top 3: Deep learning, scientific simulation, image/video processing.
  • Other big uses: cryptography, finance, bioinformatics, computer vision, rendering.
  • GPUs win because of massive parallelism, high bandwidth, mature libraries.
  • Whenever a problem is data-parallel + compute-heavy, GPU is the right answer.

Q5 — Issues in Sorting on Parallel Computers

Asked 3 times(May'23, May'24, Nov'25 — usually 6–8 marks)
How this question appeared in past papers:

(May'23 & Nov'25, 6 marks) What are the issues in sorting on parallel computers? Explain with appropriate example.

(May'24, 8 marks) What are the issues in sorting on parallel computers, explain with example?

Answer

Sorting in serial is well-understood: pick an algorithm (quicksort, mergesort, etc.) and run it. In parallel, several extra challenges arise that don't exist in the serial case.

1. Where do input elements live? Where should sorted output live?

  • In serial sorting, data sits in one CPU's memory.
  • In parallel sorting, data is DISTRIBUTED across many processors' memories.
  • Decisions to make:
    • Are inputs already distributed, or held centrally?
    • Should sorted output be on ONE processor, or distributed across all in sorted-order ranges?
  • Two common output schemes:
    • Sorted in order on processors: P0 holds smallest values, P_{p-1} holds largest.
    • Locally sorted on each processor, no global order between processors.

2. How do we compare elements on different processors?

  • Comparing two elements on the same processor is fast.
  • Comparing across processors requires communication — far slower.
  • Two primitive operations are used: compare-exchange and compare-split (see Q12).

3. Load balancing

  • Some algorithms (like quicksort) split data based on a pivot — bad pivot choice can give very uneven partitions.
  • Some processors get a tiny share and finish early; others struggle with huge shares.
  • Mitigations: pivot sampling, median-of-medians, sample sort.

4. Communication cost can dominate computation

  • Sorting is comparison-bound serially (O(n log n) work).
  • In parallel, EVERY comparison across processors involves communication.
  • For small n or slow networks, communication time exceeds computation time → no speedup.

5. Stability

  • A sort is stable if it preserves the relative order of equal elements.
  • Preserving stability is harder when data is distributed (have to track original indices).

6. Number of processors vs input size

  • If p = n (one element per processor): can use compare-exchange-based algorithms (odd-even, bitonic).
  • If p < n (multiple elements per processor): use compare-split, hybrid algorithms.
  • If p > n: extra processors sit idle.

7. Choosing the right algorithm depends on:

  • Network topology (mesh, hypercube, ring).
  • Distributed vs shared memory.
  • Available primitives (sort networks, MPI collectives).
  • Data size n vs processor count p.

8. Example — bubble sort on parallel computers

  • Sequential bubble sort is O(n²) and inherently serial (depends on previous comparisons).
  • The parallel adaptation is "odd-even transposition sort" — alternates odd-pair and even-pair compare-exchanges.
  • With n processors, it runs in n steps — speedup of n (excellent if n processors are available).
  • With p < n, we use compare-split versions and the algorithm runs in O((n/p) log² (n/p)) time.

Key points to remember

  • Issues: data distribution, comparison across processors, load balancing, communication cost.
  • Stability is harder to maintain in parallel.
  • Algorithm choice depends on n vs p, topology, and memory model.
  • Primitives: compare-exchange (p = n), compare-split (p < n).
  • Communication cost often dominates computation for sorting.

Q6 — Odd-Even Transposition Bubble Sort

Asked 3 times(May'23, May'25, Nov'25 — usually 4–8 marks)
How this question appeared in past papers:

(May'23, 8 marks) Explain odd-even transportation in bubble sort using parallel formulation. Give one stepwise example solution using odd-even transportation.

(May'25, 6 marks) Explain odd-even transportation in bubble sort using suitable example.

(Nov'25, 4 marks) Explain odd-even transportation on bubble sort using parallel formulation.

Answer
Sequential bubble sort compares each adjacent pair one at a time — slow. Odd-even transposition compares ALL odd-positioned pairs at the SAME TIME, then ALL even-positioned pairs at the SAME TIME. This way, n processors finish sorting in n phases instead of n² steps.

1. Why we need odd-even transposition

  • Sequential bubble sort is O(n²) and inherently serial (each comparison depends on the previous).
  • Odd-even transposition is the PARALLEL adaptation that breaks this dependency.
  • Uses n processors, runs in n phases, total O(n) parallel time.

2. The two alternating phases

  • Odd phase: compare pairs at positions (1,2), (3,4), (5,6), … All these pairs compare-exchange SIMULTANEOUSLY (no overlap).
  • Even phase: compare pairs at positions (0,1), (2,3), (4,5), … Again, all simultaneously.
  • Alternate odd, even, odd, even, … for n total phases.

3. Why it works (theoretical guarantee)

  • Theorem: after n alternating phases of odd-even transposition on n elements, the array is sorted.
  • n/2 odd phases + n/2 even phases = n total phases.

4. Stepwise example (n = 4 elements: [4, 2, 3, 1])

Initial:  [4, 2, 3, 1]
          P0 P1 P2 P3

Phase 1 (Odd pairs: P0-P1, P2-P3):
   Compare(4, 2) → swap.  Compare(3, 1) → swap.
   →  [2, 4, 1, 3]

Phase 2 (Even pairs: P1-P2):
   Compare(4, 1) → swap.
   →  [2, 1, 4, 3]

Phase 3 (Odd pairs: P0-P1, P2-P3):
   Compare(2, 1) → swap.  Compare(4, 3) → swap.
   →  [1, 2, 3, 4]

Phase 4 (Even pairs: P1-P2):
   Compare(2, 3) → no swap.
   →  [1, 2, 3, 4]    ✓ Sorted

5. Algorithm (compare-exchange version, one element per processor)

Algorithm ODD_EVEN_TRANSPOSITION (my_id, my_value, n):
   for step = 1 to n:
       if step is odd:
           if my_id is odd and my_id < n - 1:
               compare-exchange with neighbour (my_id + 1)
       else:   // even step
           if my_id is even and my_id < n - 1:
               compare-exchange with neighbour (my_id + 1)
   end for

6. Compare-split version (when p < n, multiple elements per processor)

  • Each processor first locally sorts its n/p elements using e.g. quicksort.
  • Then odd/even-phase compare-SPLIT operations between adjacent processors:
    • Each sends its n/p sorted elements to neighbour.
    • Both merge → 2 · n/p sorted; lower-numbered keeps smaller half, higher keeps larger half.
  • After p phases, the entire distributed array is sorted.

7. Complexity

  • With p = n (1 element per processor): Tp = O(n). Sequential bubble: O(n²). Speedup ≈ n.
  • With p < n (compare-split): Local sort: O((n/p) log(n/p)). p phases of compare-split, each O(n/p). Total: O((n/p) log(n/p) + n).

Key points to remember

  • Parallel version of bubble sort.
  • Alternates ODD-pair and EVEN-pair compare-exchanges in phases.
  • n phases total → O(n) time with n processors.
  • n/2 odd phases + n/2 even phases = n phases.
  • Compare-split version for p < n: local sort + p phases of merge-split.

Q7 — Kubernetes

Asked 3 times(May'23, May'24, Nov'25 — usually 4 marks)
How this question appeared in past papers:

(May'23, 4 marks) What is Kubernetes? Explain its features and applications.

(May'24, 4 marks) Write short note on Kubernetes.

(Nov'25, 4 marks) What is Kubernetes? Explain its features and applications.

Answer
Imagine you run 100 restaurants in different cities. You need to make sure each restaurant has the right number of cooks, that crashed cooks are replaced automatically, that customers are routed to whichever restaurant is closest, and that updates to the menu are rolled out smoothly. Kubernetes is the "manager" that does all this for containerized software applications — automatically.

1. Definition

Kubernetes (K8s) is an open-source container orchestration platform for automating deployment, scaling, and management of containerized applications across clusters of machines. Originally developed by Google, now maintained by the Cloud Native Computing Foundation (CNCF).

2. Key terminology

  • Container — a lightweight package containing an application + its dependencies (usually built with Docker).
  • Pod — the smallest deployable unit in Kubernetes; contains one or more containers sharing storage and network.
  • Node — a worker machine (VM or physical) where pods run.
  • Cluster — a set of nodes managed together.
  • Master / Control Plane — schedules pods, runs the API server and controllers.
  • Service — a stable network endpoint exposing a set of pods.
  • Deployment — declarative description of how many pods of a given type should run.
  • Namespace — virtual partition of a cluster for isolating different applications or teams.

3. Architecture diagram (text)

                ┌────────── Kubernetes Cluster ──────────┐
                │                                          │
                │    ┌────── Control Plane (master) ────┐ │
                │    │  API Server, Scheduler,           │ │
                │    │  Controller Manager, etcd         │ │
                │    └─────────────┬─────────────────────┘ │
                │                  │                       │
                │   ┌────── Worker Nodes ───────────────┐ │
                │   │  ┌─────┐  ┌─────┐  ┌─────┐         │ │
                │   │  │Node1│  │Node2│  │Node3│   ...   │ │
                │   │  │┌───┐│  │┌───┐│  │┌───┐│         │ │
                │   │  ││Pod││  ││Pod││  ││Pod││         │ │
                │   │  │└───┘│  │└───┘│  │└───┘│         │ │
                │   │  └─────┘  └─────┘  └─────┘         │ │
                │   └──────────────────────────────────────┘ │
                └────────────────────────────────────────────┘

4. Main features of Kubernetes (memorize 6–9 of these)

  1. Automatic scaling — Horizontal Pod Autoscaler adds/removes pods based on CPU/memory usage.
  2. Self-healing — automatically restarts crashed containers, replaces unhealthy pods, reschedules on failed nodes.
  3. Load balancing — distributes incoming traffic across pod replicas.
  4. Service discovery — pods can find each other via DNS or environment variables.
  5. Rolling updates and rollbacks — deploy new versions of an app without downtime; roll back if something goes wrong.
  6. Storage orchestration — auto-mount persistent volumes (cloud disks, NFS, iSCSI).
  7. Secret and configuration management — secure handling of API keys, passwords, env config.
  8. Declarative configuration — describe desired state in YAML files; K8s makes the actual state match.
  9. Multi-cloud / on-prem portability — runs on AWS, Azure, GCP, on-prem data centers.
  10. Batch execution — runs one-time jobs and scheduled CronJobs.
  11. IPv4/IPv6 dual stack, network policies for security.

5. Applications of Kubernetes

  • Microservices deployment — each microservice in its own pod; communicate via Services.
  • Machine learning pipelines — Kubeflow for distributed model training and serving.
  • Big-data processing — Spark on Kubernetes, distributed analytics.
  • CI/CD pipelines — Jenkins X, Argo CD use K8s for continuous deployment.
  • Web application hosting at scale — companies like Netflix, Spotify, Airbnb use K8s.
  • Edge computing — lightweight K3s variant for IoT and edge devices.
  • Hybrid cloud deployments — same app runs across on-prem and multiple clouds.
  • Serverless platforms — Knative, OpenFaaS run on top of K8s.

Key points to remember

  • K8s = open-source container orchestration platform (by Google, now CNCF).
  • Smallest unit = Pod; runs on Nodes; managed by Control Plane.
  • 3 features always to mention: auto-scaling, self-healing, rolling updates.
  • Other features: load balancing, service discovery, storage orchestration, declarative config.
  • Applications: microservices, ML pipelines (Kubeflow), CI/CD, big-data (Spark on K8s).

Q8 — Communication Strategies in Parallel BFS

Asked 2 times(May'25, Nov'25 — usually 5–9 marks)
How this question appeared in past papers:

(May'25, 9 marks) Write a note on: i) Random Communication Strategy ii) Ring Communication Strategy iii) Blackboard Communication Strategy.

(Nov'25, 6 marks) Explain communication strategies for parallel BFS.

Answer

When BFS runs on a distributed-memory parallel system, processors must coordinate to avoid duplicate work and to share newly-discovered nodes efficiently. Three standard communication strategies are used.

1. Random Communication Strategy

  • When a processor finds a new node, it sends the discovery to a randomly chosen processor.
  • The random target is responsible for deciding whether the node is new and adding it to its next-level frontier.
  • Pros:
    • Spreads load evenly on average.
    • Simple to implement.
    • No hotspots over time.
  • Cons:
    • May produce duplicate notifications (two processors discover the same node).
    • Unpredictable communication pattern.
    • Hard to debug.

2. Ring Communication Strategy

  • Processors are arranged in a logical ring.
  • Each processor passes newly-discovered nodes only to its immediate neighbour in the ring.
  • Discoveries propagate around the ring step by step.
  • Pros:
    • Simple, deterministic, low overhead per round.
    • Easy to implement and debug.
  • Cons:
    • Slow propagation across many processors — takes O(p) steps to reach all.
    • Higher latency for global decisions.

3. Blackboard Communication Strategy

  • A single shared "blackboard" (centralized data store) holds the current frontier.
  • Every processor reads pending work from the blackboard and writes new discoveries back.
  • Pros:
    • Clean global view of the frontier.
    • Easy duplicate elimination.
    • Simple programming model.
  • Cons:
    • The blackboard becomes a bottleneck as the number of processors grows.
    • Not scalable to thousands of processors.
    • Single point of failure.

4. Comparison table

StrategyPatternProsCons
RandomSend to random processorEven load, simpleDuplicates, unpredictable
RingPass around the ringSimple, deterministicSlow propagation (O(p))
BlackboardShared central storeEasy duplicate removalBottleneck, not scalable

Key points to remember

  • 3 strategies: Random, Ring, Blackboard.
  • Random = send to random processor; even load but duplicates.
  • Ring = pass to neighbour; deterministic but slow (O(p)).
  • Blackboard = central shared store; clean but a bottleneck.
  • Choice depends on number of processors and graph structure.

Q9 — Dijkstra's Algorithm in Parallel

Asked 1 time(May'25 — 9 marks)
How this question appeared in past papers:

(May'25, 9 marks) Explain Dijkstra's algorithm in parallel formulation.

Answer

1. Sequential Dijkstra's recap

  • Solves the Single-Source Shortest Path (SSSP) problem on a weighted graph (no negative edges).
  • Maintain a priority queue of (vertex, tentative distance).
  • Repeatedly extract the vertex with minimum distance; relax all its outgoing edges.
  • Complexity: O(V² ) with array, O((V + E) log V) with binary heap.

2. Two parallel formulations

(a) Source-partitioned formulation (All-Pairs SSSP)

  • Different processors compute SSSP from DIFFERENT source vertices simultaneously.
  • Used to compute the entire All-Pairs Shortest Path matrix.
  • Embarrassingly parallel — no communication required between processors.
  • Speedup ≈ p (ideal).
  • Useful when we need shortest paths from many sources.

(b) Source-parallel formulation (Single source, parallelized)

  • Partition the vertex set among p processors.
  • Each processor maintains tentative distances only for its assigned vertices.
  • At each iteration:
    1. Each processor finds its LOCAL minimum-distance vertex.
    2. A global all-to-one reduction picks the TRUE minimum across all processors.
    3. The winning vertex's id and distance are broadcast to all.
    4. All processors relax their local edges leading FROM that vertex.
  • Synchronize and repeat until all V vertices are settled.

3. Complexity of source-parallel formulation

  • V iterations total (one vertex settled per iteration).
  • Per iteration: local min in O(V/p) + reduction in O(log p) + edge relaxation in O(V/p).
  • Total: O(V²/p + V · log p).

4. Comparison

FormulationBest whenCommunicationTime
Source-partitioned (APSP)Many sources neededNoneO(V²/p) × p sources
Source-parallel (SSSP)One sourceGlobal reduction each iterO(V²/p + V log p)

5. Applications

  • Navigation systems (GPS routing).
  • Network routing protocols (OSPF, IS-IS).
  • Connectivity analysis in social networks.
  • Logistics and supply-chain optimization.

Key points to remember

  • Sequential Dijkstra: O(V²) array / O((V+E) log V) heap.
  • Source-partitioned: different sources on different processors (embarrassingly parallel).
  • Source-parallel: split vertices, do global min-reduction each iteration.
  • Source-parallel time: O(V²/p + V log p).

Q10 — Distributed Computing for Document Classification

Asked 1 time(May'25 — 6 marks)
How this question appeared in past papers:

(May'25, 6 marks) Explain Distributed Computing for Document Classification.

Answer

1. The problem

Given millions of documents (news articles, emails, web pages), classify each into one of several categories (sports, finance, politics, …) using a machine-learning model (Naive Bayes, SVM, neural net).

2. Why distribute?

  • A single machine cannot store or process millions of documents in reasonable time.
  • The work is embarrassingly parallel — each document can be classified independently.
  • Distributing across many nodes gives near-linear speedup.

3. MapReduce-style distributed approach

  1. Split input: millions of documents are split into chunks; one chunk per worker.
  2. Map phase: each worker loads the classification model, processes its chunk of documents, emits (category, document_id) pairs.
  3. Shuffle phase: all pairs with the same category are routed (hashed) to the same reducer.
  4. Reduce phase: for each category, the reducer assembles the list of documents belonging to it.
  5. Output: categorized results stored in a database or file system.

4. Other distributed strategies

  • Data parallelism — same model on every node, different chunks of data. Used in distributed training.
  • Model parallelism — different layers / parts of a huge model on different nodes (used for very large neural networks).
  • Pipeline parallelism — different stages of the classification pipeline (tokenize → feature extract → predict) on different nodes.

5. Frameworks used

  • Hadoop / MapReduce — classic batch processing.
  • Apache Spark — faster in-memory distributed processing.
  • Kubernetes — orchestrates classifier microservices.
  • Apache Kafka — handles real-time streams of incoming documents.
  • TensorFlow Distributed / PyTorch DDP — for distributed neural-network classifiers.

6. Applications

  • Email spam filtering at scale (Gmail).
  • News article categorization (Google News).
  • Sentiment analysis of social media at scale.
  • Search engine result ranking.
  • Content moderation on social platforms.

Key points to remember

  • Document classification is embarrassingly parallel.
  • Standard pattern: Map → Shuffle → Reduce.
  • Strategies: data parallel, model parallel, pipeline parallel.
  • Frameworks: Hadoop, Spark, Kubernetes, Kafka.
  • Applications: spam filter, news categorization, sentiment analysis.

Q11 — CRCW PRAM Formulation

Asked 1 time(May'25 — 6 marks)
How this question appeared in past papers:

(May'25, 6 marks) Write a short note on parallel formulation for CRCW PRAM.

Answer

1. What is PRAM?

PRAM (Parallel Random Access Machine) is a theoretical model of parallel computation where p processors all share a common memory. It is used to analyze the theoretical time complexity of parallel algorithms.

2. The 4 PRAM variants (based on memory access rules)

ModelReadWrite
EREWExclusive (one at a time)Exclusive
CREWConcurrent (many at once)Exclusive
ERCWExclusiveConcurrent
CRCWConcurrentConcurrent

3. CRCW PRAM — Concurrent Read Concurrent Write

  • Multiple processors can read the same memory location at the same instant.
  • Multiple processors can write to the same memory location at the same instant.
  • The MOST POWERFUL PRAM model — fastest for many algorithms.

4. How to handle concurrent writes (3 conventions)

  • Common CRCW: all processors writing to the same location must write the SAME value (else error).
  • Arbitrary CRCW: if multiple processors write, an arbitrary one's value wins.
  • Priority CRCW: the lowest-numbered processor wins.

5. Example — find maximum of n numbers on CRCW PRAM (O(1) time!)

1. Use n² processors. Pair them as (P_ij) for i, j in [0, n-1].
2. Initialize array WIN[0..n-1] to TRUE (everyone is potentially max).
3. In parallel, P_ij compares A[i] with A[j]:
       if A[i] < A[j]: set WIN[i] = FALSE
4. In parallel, scan WIN: the index where WIN[i] is still TRUE is the max.

Total time: O(1) using n² processors. On weaker PRAM models this would be O(log n).

6. CRCW PRAM formulation of Quicksort

  • All n processors read the pivot in parallel (concurrent read).
  • Each processor compares its element to the pivot.
  • Use parallel prefix-sum to compute new indices for each element.
  • All processors write their element to its new position in parallel.
  • Recurse on the two halves.
  • Complexity: O(log² n) on CRCW PRAM (compared to O(n log n) sequential).

7. Why CRCW is theoretical

  • Real hardware cannot do unlimited concurrent reads/writes for free.
  • Real systems have communication delays, cache coherence overhead, etc.
  • CRCW PRAM is useful only as a theoretical "upper bound" on what parallel hardware can achieve.

Key points to remember

  • PRAM = theoretical shared-memory parallel model.
  • 4 variants: EREW, CREW, ERCW, CRCW.
  • CRCW = Concurrent Read Concurrent Write — most powerful.
  • Conventions: Common, Arbitrary, Priority.
  • Quicksort on CRCW PRAM: O(log² n) time.
  • Useful for theoretical analysis; real systems can't match it.

Q12 — Compare-Exchange and Compare-Split Operations

Asked 1 time(Nov'25 — 8 marks)
How this question appeared in past papers:

(Nov'25, 8 marks) Explain compare exchange and compare-split operation on parallel Computers.

Answer

Compare-exchange and compare-split are the two fundamental building blocks used by almost every parallel sorting algorithm to compare data across different processors.

1. Compare-Exchange (when p = n, one element per processor)

  • Two processors Pi and Pj each hold ONE element.
  • They exchange their elements.
  • After the exchange, Pi keeps the SMALLER value, Pj keeps the LARGER.

Algorithm

Compare-Exchange (P_i, P_j):
   P_i sends its value to P_j
   P_j sends its value to P_i
   Both compare the two values:
       P_i keeps min(a, b)
       P_j keeps max(a, b)

Example

Before:  P0 has 7,  P1 has 3
Exchange: both send their values
After:   P0 has 3 (min),  P1 has 7 (max)

2. Compare-Split (when p < n, multiple elements per processor)

  • Two processors Pi and Pj each hold n/p SORTED elements.
  • They send all their data to each other.
  • Each processor now has 2 · (n/p) sorted elements (merge of both halves).
  • They merge the two sorted lists.
  • Pi keeps the SMALLER half; Pj keeps the LARGER half.

Algorithm

Compare-Split (P_i, P_j):
   P_i sends its n/p sorted elements to P_j
   P_j sends its n/p sorted elements to P_i
   Both merge the two sorted lists into one sorted list of size 2n/p
   P_i keeps the lower n/p elements
   P_j keeps the upper n/p elements

Example (n/p = 3)

Before:  P0 has [1, 4, 7],  P1 has [2, 3, 8]
After exchange + merge: combined sorted = [1, 2, 3, 4, 7, 8]
After split: P0 keeps [1, 2, 3] (lower), P1 keeps [4, 7, 8] (upper)

3. Comparison

AspectCompare-ExchangeCompare-Split
Elements per processor1n/p
Used whenp = np < n
Communication1 element each wayn/p elements each way
Local work1 comparisonMerge of two sorted lists
Used inBitonic sort, odd-even sort (1 elem/proc)Practical parallel sorts (block sorts)

4. Why both are needed

  • Compare-exchange is theoretically clean but requires n processors — often unrealistic.
  • Compare-split is the practical version when p << n.
  • Together they allow sorting algorithms to scale from fine-grained (n processors) to coarse-grained (few processors).

Key points to remember

  • Both are building blocks of parallel sorting.
  • Compare-exchange: 1 element per processor; pair keeps min/max.
  • Compare-split: n/p elements per processor; pair merges + splits into lower/upper halves.
  • Compare-exchange used when p = n; compare-split when p < n.
  • Compare-split needs O(n/p) communication; compare-exchange needs O(1).

Q13 — Parallel Quicksort (Recursive Decomposition + Shared vs Message-Passing)

Asked 2 times(Nov'25 — 8 + 6 marks)
How this question appeared in past papers:

(Nov'25, 8 marks) Explain the recursive decomposition in parallelizing quick sort.

(Nov'25, 6 marks) Compare shared address and message passing formulation of quick sort.

Answer

Part A — Recursive decomposition in parallel Quicksort

1. Quicksort recap (sequential)

  1. Pick a pivot (random, median, etc.).
  2. Partition the array: elements ≤ pivot to one side, elements > pivot to the other.
  3. Recursively sort the two sides.

Average complexity: O(n log n). Worst case: O(n²) with bad pivot.

2. Where is the parallelism?

  • The two recursive calls in step 3 work on INDEPENDENT halves.
  • They can run on DIFFERENT processors in parallel.
  • This is called recursive decomposition.

3. Algorithm (recursive decomposition)

PARALLEL_QUICKSORT (A, processors):
   if size(A) is small: sort sequentially; return
   pivot = choose pivot
   left, right = partition A using pivot
   if processors > 1:
       send right half to half of processors
       in parallel:
           recursively sort left on first half of processors
           recursively sort right on second half of processors
   else:
       sort both halves sequentially

4. Complexity

  • If the pivot perfectly splits the array each time, after log p levels of recursion, p processors are independently sorting n/p elements each.
  • Total parallel time: O(n) for partitioning + O((n/p) log(n/p)) for local sorts ≈ O(n).
  • Speedup ≈ log p (because partitioning is still serial-ish).

5. Challenges

  • BAD pivot choice → load imbalance.
  • Partition step itself is hard to parallelize efficiently.
  • Communication cost when redistributing halves.

Part B — Shared-Address vs Message-Passing formulation

Shared-Address formulation (e.g., OpenMP on multi-core CPU)

  • All processors share one common memory space.
  • The array stays in shared memory; processors use atomic counters / parallel prefix-sum to determine new positions during partitioning.
  • The pivot value is "shared" implicitly (anyone can read it).
  • No need to redistribute data — pointers into the same array.

Message-Passing formulation (e.g., MPI on cluster)

  • Each processor has its OWN private memory.
  • The pivot value must be explicitly broadcast to all processors.
  • After local partitioning, data must be physically redistributed (sent over network) so that each half goes to the right subgroup of processors.
  • Significant communication cost.

Side-by-side comparison

AspectShared-AddressMessage-Passing
MemoryOne shared address spaceEach processor has own memory
Partition stepUse atomic counters / scanData must be redistributed
Pivot broadcastImplicit (shared)Explicit (MPI_Bcast)
Load balanceEasier (work-stealing)Harder (depends on pivot)
Communication costLow (just synchronisation)High (data movement)
ScalabilityLimited by shared-memory busScales to many processors
Example platformOpenMP on multicore CPUMPI on cluster

Key points to remember

  • Recursive decomposition = parallel calls on the two halves after partitioning.
  • Pivot quality critical for load balance.
  • Shared-address: simpler, lower communication, limited scalability.
  • Message-passing: more communication, but scales to many processors.
  • OpenMP = shared-address example; MPI = message-passing example.
